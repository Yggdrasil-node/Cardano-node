Focus on wiring crates together cleanly, preserving deterministic startup and shutdown behavior, and keeping runtime concerns out of domain crates.

## Scope
- Runtime orchestration, CLI, sync lifecycle, and top-level process behavior.
- Integration of storage, consensus, ledger, mempool, and network crates.

##  Rules *Non-Negotiable*
- The node crate MUST remain an integration layer and MUST NOT absorb ledger or consensus business logic.
- Reusable peer-selection policy, tracer transports, or protocol-facing state management MUST move into an appropriate crate instead of growing inside `node`.
- Configuration, runtime startup, and sync orchestration MUST stay explicit.
- Composition MUST be preferred over cross-crate shortcuts.
- Major runtime entry points MUST have smoke coverage.
- Public node-facing integration types and runtime helpers MUST have Rustdocs when startup, shutdown, configuration, or sync semantics are not obvious.
- Stay true to the official type naming and terminology for node concepts, network protocols, and ledger types when possible.
- Integration behavior MUST always be explained by anchoring it in the official node and the relevant upstream IntersectMBO implementation.
- Always read the folder specific `**/AGENTS.md` files. They MUST stay current and MUST remain operational rather than long-form documentation. If the folder context is outdated, missing, or incorrect, update the relevant AGENTS.md file.

## Current Phase
- Keep the node crate thin and integration-focused.
- **Configuration**: `NodeConfigFile` (JSON, serde) with primary peer address, ordered `bootstrap_peers`, richer topology fields (`local_roots`, `public_roots`, `use_ledger_after_slot`, `peer_snapshot_file`), durable `storage_dir`, network magic, protocol versions, KES parameters, keepalive interval, and upstream-aligned tracing fields (`TurnOnLogging`, `UseTraceDispatcher`, `TraceOptionNodeName`, `TraceOptions`, `TraceOptionForwarder`, `TraceOptionMetricsPrefix`, `TraceOptionResourceFrequency`). `default_config()` returns mainnet defaults. `NetworkPreset` enum (`Mainnet | Preprod | Preview`) with `FromStr`/`Display` and per-network constructors (`mainnet_config()`, `preprod_config()`, `preview_config()`). `NodeConfigFile.topology_config()` now reconstructs the typed network-owned topology view, `parse_peer_snapshot_json()`/`load_peer_snapshot_file()` read upstream `peerSnapshotFile` JSON into a normalized `LedgerPeerSnapshot`, and `eligible_ledger_fallback_peers()` uses `yggdrasil-network` policy helpers to expose only currently eligible, deduplicated ledger fallbacks after `useLedgerAfterSlot`, latest-slot, ledger-state, and peer-snapshot freshness gating.
- **Configuration**: `NodeConfigFile` (JSON, serde) with primary peer address, ordered `bootstrap_peers`, richer topology fields (`local_roots`, `public_roots`, `use_ledger_after_slot`, `peer_snapshot_file`), durable `storage_dir`, network magic, protocol versions, KES parameters, keepalive interval, and upstream-aligned tracing fields (`TurnOnLogging`, `UseTraceDispatcher`, `TraceOptionNodeName`, `TraceOptions`, `TraceOptionForwarder`, `TraceOptionMetricsPrefix`, `TraceOptionResourceFrequency`). `default_config()` returns mainnet defaults. `NetworkPreset` enum (`Mainnet | Preprod | Preview`) with `FromStr`/`Display` and per-network constructors (`mainnet_config()`, `preprod_config()`, `preview_config()`). `NodeConfigFile.topology_config()` now reconstructs the typed network-owned topology view, `parse_peer_snapshot_json()`/`load_peer_snapshot_file()` read upstream `peerSnapshotFile` JSON into a normalized `LedgerPeerSnapshot`, and `eligible_ledger_fallback_peers()` uses `yggdrasil-network` policy helpers to expose only currently eligible, deduplicated ledger fallbacks after `useLedgerAfterSlot`, latest-slot, ledger-state, and peer-snapshot freshness gating. `NodeConfigFile::load_plutus_cost_model()` now reads `alonzo-genesis.json`, derives a simplified flat CEK model from the upstream named `costModels.PlutusV1` map, and feeds that into verified sync config for checkpoint-tracked phase-2 validation.
- Extraction rule: keep JSON config loading and preset resolution in `node`, but once topology handling needs persistent peer state, ledger-peer discovery, peer sharing, or governor-style decision logic, move that behavior into `crates/network` rather than extending `main.rs`/`runtime.rs`/`config.rs` further.
- **CLI**: `clap`-based binary with `run` (connect + sync) and `default-config` (emit JSON) subcommands. CLI flags (`--peer`, `--network-magic`, `--no-verify`, `--batch-size`, `--checkpoint-interval-slots`, `--max-ledger-snapshots`, `--checkpoint-trace-max-frequency`, `--checkpoint-trace-severity`, `--checkpoint-trace-backend`, `--network`) override config-file values. `--network` accepts `mainnet`, `preprod`, or `preview` as a preset.
- **Network config files**: `node/configuration/{mainnet,preprod,preview}/` each contain byron-genesis.json, shelley-genesis.json, alonzo-genesis.json, conway-genesis.json, config.json, topology.json sourced from the Cardano Operations Book. Preset peer ordering is derived from the vendored `topology.json` files by preferring bootstrap peers first, then trustable local roots, then other local roots, then public roots. Startup now also recovers the immutable-ledger view, extracts directly dialable registered pool relays, merges them with any `peerSnapshotFile` peers resolved relative to the config file, derives slot-based freshness from the recovered ChainDb tip, and appends only policy-eligible ledger or snapshot peers to the fallback list. Genesis-derived protocol parameters are now loaded once into a shared base `LedgerState` before any recovery path runs, so startup peer selection, `validate-config`, `status`, and resumed sync all replay storage from the same genesis-seeded baseline. Coordinated-storage reconnect runners now refresh both checkpoint-derived ledger pool relays and the resolved `peerSnapshotFile` peer set before each bootstrap attempt when `useLedgerPeers` is enabled, so reconnects are not limited to the static startup fallback list.
- Runtime bootstrap wiring is implemented (`NodeConfig`, `PeerSession`, `bootstrap`, `bootstrap_with_fallbacks`) with smoke coverage.
- Full sync orchestration stack is implemented: `sync_step`, `sync_steps`, typed decode bridges, bounded loops, intersection finding, batch apply, managed sync service with graceful shutdown via `tokio::signal::ctrl_c`. The CLI now uses a reconnecting verified sync runner that re-bootstraps through configured peers on ChainSync or BlockFetch connectivity loss.
- **Tracing**: `node/src/tracer.rs` provides a thin node-side trace dispatcher that interprets `TraceOptions` and emits human- or machine-formatted runtime trace objects for bootstrap, reconnect, sync progress, shutdown, fatal runtime failures, and checkpoint cadence decisions. The dispatcher now enforces `maxFrequency` rate limits per namespace. Default `TraceOptions` include a dedicated `Node.Recovery.Checkpoint` namespace override at `Info` with `maxFrequency: 1.0` so checkpoint events can be tuned independently of the root logger without flooding stdout during long sync runs. This preserves the official node/tracer producer-consumer split while keeping local stdout tracing usable before dedicated tracer transport is implemented.
- Multi-era block decode (`MultiEraBlock`, `decode_multi_era_block`, `decode_multi_era_blocks`) with Byron opaque, Shelley/Allegra/Mary/Alonzo decoded as `ShelleyBlock`, Babbage decoded as `BabbageBlock`, and Conway decoded as `ConwayBlock` is implemented. All seven era tags (0–7) are handled.
- Consensus header verification bridge (`verify_shelley_header`, `verify_multi_era_block`, `VerificationConfig`) is wired into the sync flow.
- Block body hash verification (`verify_block_body_hash`, `VerificationConfig.verify_body_hash`) computes Blake2b-256 of block body elements and compares against the header-declared hash. Wired into verified batch fetch before storage apply.
- Block header hash computation uses real Blake2b-256.
- Mempool sync eviction (`extract_tx_ids`, `evict_confirmed_from_mempool`) is implemented. TxSubmission runtime integration now includes `serve_txsubmission_request_from_mempool` for one-shot fee-ordered request handling, plus `serve_txsubmission_request_from_reader`/`run_txsubmission_service` for upstream-style managed outbound serving using a `TxSubmissionMempoolReader`, `MempoolSnapshot`, and monotonic `last_idx` cursor. `run_txsubmission_service_shared` provides the same managed outbound flow over `SharedMempool` so concurrent mempool updates can be observed while the service is running. `add_tx_to_mempool`/`add_tx_to_shared_mempool` now provide explicit addTx-style inbound admission helpers, and `add_txs_to_mempool`/`add_txs_to_shared_mempool` provide ordered addTxs-style admission where accepted transactions advance the staged `LedgerState` for later transactions in the same batch. All paths relay stored submitted-transaction bytes and send `MsgDone` on blocking requests when no eligible transactions remain.
- Verified sync service (`run_verified_sync_service`, `VerifiedSyncServiceConfig`, `VerifiedSyncServiceOutcome`) uses the multi-era verified pipeline with per-block nonce evolution tracking and optional ChainState tracking. `VerifiedSyncServiceConfig` now also carries `LedgerCheckpointPolicy` (`min_slot_delta`, `max_snapshots`) so coordinated storage can throttle checkpoint writes and prune old snapshots deterministically. Verified execution is split into fetch-only `sync_batch_verified` plus a ChainDb apply phase so coordinated storage mutation happens after network I/O completes. `run_verified_sync_service_chaindb` extends the verified flow across `yggdrasil-storage::ChainDb`, promotes stable entries, refreshes typed checkpoints during verified sync, and applies retention on the ledger snapshot store. Runtime also exposes `run_reconnecting_verified_sync_service_chaindb`, `run_reconnecting_verified_sync_service_chaindb_with_tracer`, `resume_reconnecting_verified_sync_service_chaindb`, and shared-ChainDb resume variants so reconnecting sync can keep using peer fallback logic while coordinated storage owns stable promotion and recovery. The traced ChainDb runtime path now emits `Node.Recovery.Checkpoint` events for checkpoint persists, skips, and store clears so operators can observe checkpoint cadence and pruning decisions during live sync. CLI `run` now uses the coordinated resume path with file-backed storage under `storage_dir` by default, and `NodeConfigFile` exposes `checkpoint_interval_slots`, `max_ledger_snapshots`, and optional `inbound_listen_addr` to control live checkpoint cadence and inbound node-to-node serving.
- Low-level reconnect API note: `ReconnectingVerifiedSyncRequest` now includes `base_ledger_state` and the coordinated-storage reconnect path uses that state for checkpoint replay instead of hardcoding `LedgerState::new(Era::Byron)`. Callers that need genesis-derived validation thresholds in low-level reconnect tests/helpers must pass a genesis-seeded base state explicitly.
- Coordinated-storage recovery now includes `recover_ledger_state_chaindb`, which restores the latest typed checkpoint and replays both immutable and volatile blocks into a `LedgerState`. The coordinated reconnecting runtime paths now both persist refreshed typed checkpoints after successful verified batches, so restart recovery is fed by either the direct ChainDb reconnecting runner or the explicit resumed runner.
- ChainState integration: `multi_era_block_to_chain_entry`, `track_chain_state_entries`, `track_chain_state`, `promote_stable_blocks`, and `promote_stable_blocks_chaindb` wire consensus `ChainState` into both the legacy split-store flow and the newer coordinated storage path. `VerifiedSyncServiceConfig.security_param` enables chain tracking with stability window enforcement.
- Genesis parameters in `NodeConfigFile`: `epoch_length` (432000), `security_param_k` (2160), `active_slot_coeff` (0.05). Stability window computed as `3k/f`.
- Prefer smokeable runtime wiring over feature-rich operational behavior at this stage.
- **Inbound server session**: `node/src/server.rs` provides `InboundPeerSession` (wraps ChainSyncServer, BlockFetchServer, KeepAliveServer, TxSubmissionServer for an accepted inbound peer), `run_keepalive_server` (echo loop), `run_blockfetch_server` (storage-backed via `BlockProvider` trait), `run_chainsync_server` (storage-backed via `ChainProvider` trait), `run_txsubmission_server` (pulls submitted transactions from the remote peer via TxSubmission2), and `run_inbound_accept_loop` (listener loop that accepts connections and spawns per-peer server tasks with optional `Arc<dyn BlockProvider>` + `Arc<dyn ChainProvider>` + `Arc<dyn TxSubmissionConsumer>`). `SharedChainDb<I,V,L>` implements `BlockProvider` and `ChainProvider` over `Arc<RwLock<ChainDb>>` with CBOR point decode, `(from, to]` BlockFetch range handling across immutable+volatile storage, and ChainSync cursor tracking by encoded `Point` rather than raw header bytes. ChainSync roll-forward headers are re-derived from stored `Block.raw_cbor` rather than serving full block payloads. `Block.raw_cbor: Option<Vec<u8>>` carries original wire bytes through the sync pipeline for BlockFetch re-serving. `MultiEraSyncStep::RollForward` carries `raw_blocks: Option<Vec<Vec<u8>>>`; verified fetch preserves raw bytes before the apply phase stores them. `SharedTxSubmissionConsumer` now recovers the current ledger snapshot from coordinated storage, decodes submitted transactions against the current ledger era, and admits them into a shared mempool with the existing runtime helper. `main.rs` now optionally binds `PeerListener` from `NodeConfigFile.inbound_listen_addr`, runs inbound serving alongside sync, and shares the coordinated ChainDb through the shared resume runtime path without holding the write lock across network fetches.
- **Governor runtime loop**: `config.rs` now carries explicit governor knobs (`governor_tick_interval_secs`, `governor_target_known`, `governor_target_established`, `governor_target_active`) with serde defaults and preset values. `runtime.rs` exposes `RuntimeGovernorConfig`, `seed_peer_registry()`, `local_root_targets_from_config()`, and `run_governor_loop()` so `main.rs` can seed a `PeerRegistry` from the primary peer and topology-owned root sources, then spawn a shutdown-aware periodic governor task beside inbound serving and sync. The current loop now owns an `OutboundPeerManager`: `PromoteToWarm` dials and retains a negotiated `PeerSession`, warm peers receive periodic KeepAlive traffic using the configured interval, `DemoteToCold` aborts the mux, DNS-backed root sources refresh through `RootPeerProviderState`, and ledger peers are re-derived from shared `ChainDb` recovery plus the optional peer snapshot file through the network crate's existing policy reconciliation helpers. Next work is wiring hot peers into real multi-peer sync/data-protocol scheduling.
