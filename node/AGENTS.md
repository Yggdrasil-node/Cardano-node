# Guidance for runtime orchestration, CLI, and integration work
Focus on wiring crates together cleanly, preserving deterministic startup and shutdown behavior, and keeping runtime concerns out of domain crates.

## Scope
- Runtime orchestration, CLI, sync lifecycle, and top-level process behavior.
- Integration of storage, consensus, ledger, mempool, and network crates.

##  Rules *Non-Negotiable*
- The node crate MUST remain an integration layer and MUST NOT absorb ledger or consensus business logic.
- Reusable peer-selection policy, tracer transports, or protocol-facing state management MUST move into an appropriate crate instead of growing inside `node`.
- Configuration, runtime startup, and sync orchestration MUST stay explicit.
- Composition MUST be preferred over cross-crate shortcuts.
- Major runtime entry points MUST have smoke coverage.
- Public node-facing integration types and runtime helpers MUST have Rustdocs when startup, shutdown, configuration, or sync semantics are not obvious.
- Stay true to the official type naming and terminology for node concepts, network protocols, and ledger types when possible.
- Integration behavior MUST always be explained by anchoring it in the official node and the relevant upstream IntersectMBO implementation.
- Always read the folder specific `**/AGENTS.md` files. They MUST stay current and MUST remain operational rather than long-form documentation. If the folder context is outdated, missing, or incorrect, update the relevant `AGENTS.md` file.

## Current Phase
- Keep the node crate thin and integration-focused.
- **Connection manager runtime integration (Step 1.2)**: `main.rs` now creates a shared `ConnectionManagerState` and passes it to both `run_governor_loop()` and `run_inbound_accept_loop()`. Outbound `PromoteToWarm`/`DemoteToCold` flows are now gated by CM transitions (`acquire_outbound_connection` + `outbound_handshake_done` and `release_outbound_connection`), with returned `CmAction`s executed deterministically in runtime order.
- **Inbound CM tracking (Step 1.2)**: `run_inbound_accept_loop()` now optionally wires accepted peers through `include_inbound_connection` + `inbound_handshake_done` and calls `release_inbound_connection` on session teardown, so inbound accept no longer bypasses CM state tracking.
- **Inbound governor runtime integration (Step 1.3)**: `main.rs` now creates a shared `InboundGovernorState` and passes it into inbound serving plus peer sharing. `run_inbound_accept_loop()` drives IG events for new connections, responder activity deltas, periodic inactivity/maturation ticks, and mux-finished teardown; IG actions are bridged into CM (`promoted_to_warm_remote`, `demoted_to_cold_remote`, `release_inbound_connection` + `apply_commit_result`). `SharedPeerSharingProvider` can now include IG mature duplex peers in outbound shared-peer responses.
- **Control-message wiring (Step 1.4)**: `runtime.rs` now tracks per-peer `TemperatureBundle<ControlMessage>` in `ManagedWarmPeer` and applies lifecycle transitions on promote/activate/deactivate/close (`Continue`/`Quiesce`/`Terminate`) instead of relying only on hot/warm status flags.
- **Churn + judgement feedback wiring (Steps 1.5/1.6)**: `run_governor_loop()` now derives `LedgerStateJudgement` each tick from coordinated recovery and feeds `peer_selection_mode()`, `compute_association_mode()`, and churn regime selection (`fetch_mode_from_judgement`, `churn_mode_from_fetch_mode`, `pick_churn_regime`) before calling `GovernorState::tick()`.
- **Peer metrics wiring (Step 1.7)**: runtime now records governor peer metrics from live hot-peer observations (`record_upstreamyness` during tip refresh and `record_fetchyness` for selected best hot peers), and traces per-action metric score from `PeerMetrics::combined_score()`.
- **Connection-manager timeout + outbound drain enforcement (Step 1.8)**: governor runtime now executes `ConnectionManagerState::timeout_tick()` each loop and applies emitted `CmAction`s, activating CM responder/time-wait timeout transitions during normal operation. On shutdown it also releases CM-managed warm/hot peers via `release_outbound_connection` before stopping.
- **Configuration**: `NodeConfigFile` (serde shape aligned to upstream config keys) with primary peer address, ordered `bootstrap_peers`, richer topology fields (`local_roots`, `public_roots`, `use_ledger_after_slot`, `peer_snapshot_file`), `topology_file_path` (`TopologyFilePath` config key for external topology file loading), durable `storage_dir`, network magic, protocol versions, KES parameters, keepalive interval, and upstream-aligned tracing fields (`TurnOnLogging`, `UseTraceDispatcher`, `TraceOptionNodeName`, `TraceOptions`, `TraceOptionForwarder`, `TraceOptionMetricsPrefix`, `TraceOptionResourceFrequency`). `load_topology_file()` reads upstream P2P JSON format and `apply_topology_to_config()` overrides inline topology from external file. `load_effective_config` accepts JSON first with YAML fallback for the same config schema. `default_config()` returns mainnet defaults. `NetworkPreset` enum (`Mainnet | Preprod | Preview`) with `FromStr`/`Display` and per-network constructors (`mainnet_config()`, `preprod_config()`, `preview_config()`). `NodeConfigFile.topology_config()` now reconstructs the typed network-owned topology view, `parse_peer_snapshot_json()`/`load_peer_snapshot_file()` read upstream `peerSnapshotFile` JSON into a normalized `LedgerPeerSnapshot`, and `eligible_ledger_fallback_peers()` uses `yggdrasil-network` policy helpers to expose only currently eligible, deduplicated ledger fallbacks after `useLedgerAfterSlot`, latest-slot, ledger-state, and peer-snapshot freshness gating. `NodeConfigFile::load_plutus_cost_model()` now reads `alonzo-genesis.json`, derives per-builtin parameterized CEK costing from upstream named `costModels.PlutusV1`/`PlutusV2` maps (with flat fallback for unmapped builtins), and when those named maps are absent it falls back to the live 251-entry Conway `plutusV3CostModel` ordered array instead of the older CEK-only structural mapping.
- Extraction rule: keep JSON config loading and preset resolution in `node`, but once topology handling needs persistent peer state, ledger-peer discovery, peer sharing, or governor-style decision logic, move that behavior into `crates/network` rather than extending `main.rs`/`runtime.rs`/`config.rs` further.
- **CLI**: `clap`-based binary with `run` (connect + sync) and `default-config` (emit JSON) subcommands. CLI flags (`--peer`, `--network-magic`, `--no-verify`, `--batch-size`, `--checkpoint-interval-slots`, `--max-ledger-snapshots`, `--checkpoint-trace-max-frequency`, `--checkpoint-trace-severity`, `--checkpoint-trace-backend`, `--network`, `--topology`, `--database-path`, `--port`, `--host-addr`) override config-file values. `--network` accepts `mainnet`, `preprod`, or `preview` as a preset. `--topology` loads an external P2P topology file (upstream JSON format) and overrides inline topology config; `TopologyFilePath` config key provides the same capability without the CLI flag. `--database-path` overrides `storage_dir`. `--port` and `--host-addr` override the inbound listen address on `run`.
- **Network config files**: `node/configuration/{mainnet,preprod,preview}/` each contain byron-genesis.json, shelley-genesis.json, alonzo-genesis.json, conway-genesis.json, config.json, topology.json sourced from the Cardano Operations Book. Preset peer ordering is derived from the vendored `topology.json` files by preferring bootstrap peers first, then trustable local roots, then other local roots, then public roots. Startup now also recovers the immutable-ledger view, extracts directly dialable registered pool relays, merges them with any `peerSnapshotFile` peers resolved relative to the config file, derives slot-based freshness from the recovered ChainDb tip, and appends only policy-eligible ledger or snapshot peers to the fallback list. Genesis-derived protocol parameters are now loaded once into a shared base `LedgerState` before any recovery path runs, so startup peer selection, `validate-config`, `status`, and resumed sync all replay storage from the same genesis-seeded baseline. Coordinated-storage reconnect runners now refresh both checkpoint-derived ledger pool relays and the resolved `peerSnapshotFile` peer set before each bootstrap attempt when `useLedgerPeers` is enabled, so reconnects are not limited to the static startup fallback list.
- Runtime bootstrap wiring is implemented (`NodeConfig`, `PeerSession`, `bootstrap`, `bootstrap_with_fallbacks`) with smoke coverage.
- Full sync orchestration stack is implemented: `sync_step`, `sync_steps`, typed decode bridges, bounded loops, intersection finding, batch apply, managed sync service with graceful shutdown via `tokio::signal::ctrl_c`. The CLI now uses a reconnecting verified sync runner that re-bootstraps through configured peers on ChainSync or BlockFetch connectivity loss.
- Diffusion pipelining (DPvDV) tentative-header wiring is now active in node orchestration: `main.rs` allocates shared `TentativeState`, inbound ChainSync serving reads it via `SharedChainDb::from_arc_with_tentative`, and reconnecting verified sync paths carry it through `with_tentative_state(...)` into `sync_batch_verified_with_tentative()`, which sets tentative headers before body validation and clears adopted/trap outcomes. `run_chainsync_server()` now rolls followers back when a served tentative header is trapped.
- **Tracing**: `node/src/tracer.rs` provides a thin node-side trace dispatcher that interprets `TraceOptions` and emits human- or machine-formatted runtime trace objects for bootstrap, reconnect, sync progress, shutdown, fatal runtime failures, and checkpoint cadence decisions. The dispatcher now enforces severity hierarchy threshold filtering plus longest-prefix namespace matching for severity/backends/`maxFrequency`/detail. `TraceDetail` enum (`DMinimal`/`DNormal`/`DDetailed`/`DMaximum`) mirrors upstream `Cardano.Logging.Types.DetailLevel` and is resolved per-namespace with longest-prefix matching via `detail_for()`. `TraceBackend::StdoutHumanColoured` emits ANSI-coloured lines (severity-keyed colour palette matching upstream HumanFormatColoured). `backends_for()` recognises all upstream backend strings (`EKGBackend`, `Forwarder`, `PrometheusSimple …`, `Stdout HumanFormatColoured`) so mainnet config files parse without silent backend drops. `trace_runtime_detailed()` gates events on a minimum detail level. Default `TraceOptions` include a dedicated `Node.Recovery.Checkpoint` namespace override at `Info` with `maxFrequency: 1.0` so checkpoint events can be tuned independently of the root logger without flooding stdout during long sync runs. This preserves the official node/tracer producer-consumer split while keeping local stdout tracing usable before dedicated tracer transport is implemented.
- Multi-era block decode (`MultiEraBlock`, `decode_multi_era_block`, `decode_multi_era_blocks`) with Byron opaque, Shelley/Allegra/Mary/Alonzo decoded as `ShelleyBlock`, Babbage decoded as `BabbageBlock`, and Conway decoded as `ConwayBlock` is implemented. All seven era tags (0–7) are handled.
- Consensus header verification bridge (`verify_shelley_header`, `verify_multi_era_block`, `VerificationConfig`) is wired into the sync flow. `verify_multi_era_block` now also performs BBODY-level `validate_block_protocol_version` (era/version consistency) before header verification. `validate_block_body_size` checks declared vs actual serialized body size (upstream `WrongBlockBodySizeBBODY`). Both are peer-attributable errors.
- Block body hash verification (`verify_block_body_hash`, `VerificationConfig.verify_body_hash`) computes Blake2b-256 of block body elements and compares against the header-declared hash. Wired into verified batch fetch before storage apply.
- Block header hash computation uses real Blake2b-256.
- Mempool sync eviction (`extract_tx_ids`, `evict_confirmed_from_mempool`) is implemented and now wired into reconnecting sync loops: the reconnecting verified sync request carries `mempool: Option<SharedMempool>` and after each batch both ChainDb inner functions (shared and non-shared) extract confirmed tx IDs from `RollForward` steps and call `remove_confirmed()` + `purge_expired()` on the mempool, tracing eviction counts under `Mempool.Eviction`. TxSubmission runtime integration now includes `serve_txsubmission_request_from_mempool` for one-shot fee-ordered request handling, plus `serve_txsubmission_request_from_reader`/`run_txsubmission_service` for upstream-style managed outbound serving using a `TxSubmissionMempoolReader`, `MempoolSnapshot`, and monotonic `last_idx` cursor. `run_txsubmission_service_shared` provides the same managed outbound flow over `SharedMempool` so concurrent mempool updates can be observed while the service is running. `add_tx_to_mempool`/`add_tx_to_shared_mempool` now provide explicit addTx-style inbound admission helpers, and `add_txs_to_mempool`/`add_txs_to_shared_mempool` provide ordered addTxs-style admission where accepted transactions advance the staged `LedgerState` for later transactions in the same batch. All paths relay stored submitted-transaction bytes and send `MsgDone` on blocking requests when no eligible transactions remain.
- Verified sync service (`run_verified_sync_service`, `VerifiedSyncServiceConfig`, `VerifiedSyncServiceOutcome`) uses the multi-era verified pipeline with per-block nonce evolution tracking and optional ChainState tracking. `VerifiedSyncServiceConfig` now also carries `LedgerCheckpointPolicy` (`min_slot_delta`, `max_snapshots`) so coordinated storage can throttle checkpoint writes and prune old snapshots deterministically. Verified execution is split into fetch-only `sync_batch_verified` plus a ChainDb apply phase so coordinated storage mutation happens after network I/O completes. `run_verified_sync_service_chaindb` extends the verified flow across `yggdrasil-storage::ChainDb`, promotes stable entries, refreshes typed checkpoints during verified sync, and applies retention on the ledger snapshot store. Runtime also exposes `run_reconnecting_verified_sync_service_chaindb`, `run_reconnecting_verified_sync_service_chaindb_with_tracer`, `resume_reconnecting_verified_sync_service_chaindb`, and shared-ChainDb resume variants so reconnecting sync can keep using peer fallback logic while coordinated storage owns stable promotion and recovery. The traced ChainDb runtime path now emits `Node.Recovery.Checkpoint` events for checkpoint persists, skips, and store clears so operators can observe checkpoint cadence and pruning decisions during live sync. CLI `run` now uses the coordinated resume path with file-backed storage under `storage_dir` by default, and `NodeConfigFile` exposes `checkpoint_interval_slots`, `max_ledger_snapshots`, and optional `inbound_listen_addr` to control live checkpoint cadence and inbound node-to-node serving.
- Low-level reconnect API note: `ReconnectingVerifiedSyncRequest` now includes `base_ledger_state` and the coordinated-storage reconnect path uses that state for checkpoint replay instead of hardcoding `LedgerState::new(Era::Byron)`. Callers that need genesis-derived validation thresholds in low-level reconnect tests/helpers must pass a genesis-seeded base state explicitly.
- Coordinated-storage recovery now includes `recover_ledger_state_chaindb`, which restores the latest typed checkpoint and replays both immutable and volatile blocks into a `LedgerState`. The coordinated reconnecting runtime paths now both persist refreshed typed checkpoints after successful verified batches, so restart recovery is fed by either the direct ChainDb reconnecting runner or the explicit resumed runner.
- ChainState integration: `multi_era_block_to_chain_entry`, `track_chain_state_entries`, `track_chain_state`, `promote_stable_blocks`, and `promote_stable_blocks_chaindb` wire consensus `ChainState` into both the legacy split-store flow and the newer coordinated storage path. `VerifiedSyncServiceConfig.security_param` enables chain tracking with stability window enforcement.
- Genesis parameters in `NodeConfigFile`: `epoch_length` (432000), `security_param_k` (2160), `active_slot_coeff` (0.05). Stability window computed as `3k/f`.
- Pool performance tracking: `LedgerCheckpointTracking` accumulates per-pool block counts (`pool_block_counts: BTreeMap<PoolKeyHash, u64>`), and `advance_ledger_with_epoch_boundary` hashes each block's issuer VKey to a `PoolKeyHash`, then at epoch transitions calls `compute_pool_performance()` to derive real `UnitInterval` performance ratios from the `set` stake snapshot for `apply_epoch_boundary()`. Counts reset after each epoch boundary and on rollback.
- Prefer smokeable runtime wiring over feature-rich operational behavior at this stage.
- **Inbound server session**: `node/src/server.rs` provides `InboundPeerSession` (wraps ChainSyncServer, BlockFetchServer, KeepAliveServer, TxSubmissionServer for an accepted inbound peer), `run_keepalive_server` (echo loop), `run_blockfetch_server` (storage-backed via `BlockProvider` trait), `run_chainsync_server` (storage-backed via `ChainProvider` trait), `run_txsubmission_server` (pulls submitted transactions from the remote peer via TxSubmission2), and `run_inbound_accept_loop` (listener loop that accepts connections and spawns per-peer server tasks with optional `Arc<dyn BlockProvider>` + `Arc<dyn ChainProvider>` + `Arc<dyn TxSubmissionConsumer>`). `SharedChainDb<I,V,L>` implements `BlockProvider` and `ChainProvider` over `Arc<RwLock<ChainDb>>` with CBOR point decode, `(from, to]` BlockFetch range handling across immutable+volatile storage, and ChainSync cursor tracking by encoded `Point` rather than raw header bytes. ChainSync roll-forward headers are re-derived from stored `Block.raw_cbor` rather than serving full block payloads. `Block.raw_cbor: Option<Vec<u8>>` carries original wire bytes through the sync pipeline for BlockFetch re-serving. `MultiEraSyncStep::RollForward` carries both `raw_blocks: Vec<Vec<u8>>` (parallel to `blocks`, populated by verified fetch before the apply phase stores them) and `block_spans: Vec<BlockTxRawSpans>` (parallel pre-extracted CBOR byte spans computed once via `extract_spans_per_block` at step construction so all three roll-forward consumers — `extract_tx_ids`, `apply_multi_era_step_to_volatile`, `advance_ledger_state_with_progress` — share the cache instead of re-walking the block CBOR). `SharedTxSubmissionConsumer` now recovers the current ledger snapshot from coordinated storage, decodes submitted transactions against the current ledger era, and admits them into a shared mempool with the existing runtime helper. `main.rs` now optionally binds `PeerListener` from `NodeConfigFile.inbound_listen_addr`, runs inbound serving alongside sync, and shares the coordinated ChainDb through the shared resume runtime path without holding the write lock across network fetches.
- **Governor runtime loop**: `config.rs` now carries explicit governor knobs (`governor_tick_interval_secs`, `governor_target_known`, `governor_target_established`, `governor_target_active`) with serde defaults and preset values. `runtime.rs` exposes `RuntimeGovernorConfig`, `seed_peer_registry()`, `local_root_targets_from_config()`, and `run_governor_loop()` so `main.rs` can seed a `PeerRegistry` from the primary peer and topology-owned root sources, then spawn a shutdown-aware periodic governor task beside inbound serving and sync. The current loop now owns an `OutboundPeerManager`: `PromoteToWarm` dials and retains a negotiated `PeerSession`, warm peers receive periodic KeepAlive traffic using the configured interval, `DemoteToCold` aborts the mux, DNS-backed root sources refresh through `RootPeerProviderState`, and ledger peers are re-derived from shared `ChainDb` recovery plus the optional peer snapshot file through the network crate's existing policy reconciliation helpers using the same caller-supplied genesis-seeded `base_ledger_state` as the sync runtime. Hot-peer tip refresh now writes per-peer `hot_tip_slot` metadata into `PeerRegistry`, and reconnecting sync prefers `preferred_hot_peer()` (highest known hot tip) when biasing bootstrap attempts. Next work is wiring hot peers into real multi-peer sync/data-protocol scheduling.
- **Block production (Phase 2 — runtime loop integration)**: `node/src/block_producer.rs` now includes upstream-style `ShouldForge` flow (`check_should_forge()`), `SlotClock`, mempool body selection (`assemble_block_body()`), full forge composition (`forge_block()`), and storage conversion (`forged_block_to_storage_block()`). Forged blocks are serialized into multi-era CBOR and stored in `Block.raw_cbor` for downstream BlockFetch/ChainSync relay. `runtime.rs` now exposes `run_block_producer_loop()` and `RuntimeBlockProducerConfig`: a shutdown-aware 1-second slot loop that derives block context from ChainDb tip, runs per-slot leadership checks, forges blocks from fee-ordered mempool snapshots, inserts forged blocks into volatile ChainDb, removes included txs from mempool, updates metrics, emits `Node.BlockProduction` traces, and wakes ChainSync followers through shared tip notifications. `main.rs` now spawns this loop when block-producer credentials are configured and active slot coefficient is available, and reconnecting sync also wakes tip followers after successful batch apply. Block producer credential loading now requires `ShelleyOperationalCertificateIssuerVkey` / `--shelley-operational-certificate-issuer-vkey`, validates OpCert signature against that cold issuer key, and forges headers with the configured issuer key.
- **Block production self-validation parity**: `forge_block()` now computes `block_body_hash` and `block_body_size` from canonical serialized Conway body bytes (all elements after the header), and forged header hash now follows on-wire Praos header CBOR hashing. `run_block_producer_loop()` now runs `self_validate_forged_block()` before persistence so local forged blocks failing protocol-version/body-hash/body-size/header-identity checks are rejected before `add_volatile_block()`.
- **Runtime parity hardening (P1-P3)**: block production now performs post-forge adoption checks and emits adoption/non-adoption trace outcomes aligned with upstream `NodeKernel.forkBlockForging`; reconnecting verified sync now classifies peer-attributable validation failures as reconnect-and-punish with `ChainDB.AddBlockEvent.InvalidBlock` tracing (upstream `InvalidBlockPunishment` intent); and far-future header checks are enforced via consensus `ClockSkew` + `FutureSlotJudgement` (`InFutureCheck`) and surfaced as `SyncError::BlockFromFuture`.
- **Dynamic block-producer nonce/sigma**: `run_block_producer_loop()` now reads live `epoch_nonce` and pool `sigma` from `SharedBlockProducerState` (falling back to static config values when not yet populated). The reconnecting sync pipeline pushes `update_bp_state_nonce()` after every verified batch and `update_bp_state_sigma()` after epoch-boundary stake snapshot rotation. `main.rs` computes `bp_pool_key_hash` (Blake2b-224 of issuer cold key) at startup and threads it alongside the shared state into the sync request via `.with_bp_state()`. Reference: upstream `forkBlockForging` reads per-epoch `IndividualPoolStake` and evolving nonce.
- **PeerSharing governor-driven dispatch**: `GovernorAction::ShareRequest(peer)` handler in `runtime.rs` now dispatches real PeerSharing protocol to the specific targeted peer (looks up warm peers, calls `share_peers(amount).await`, merges results into `PeerRegistry`), replacing the previous stub that only marked the action complete. Removed non-upstream blanket `refresh_peer_share_sources()` fan-out. Reference: `Ouroboros.Network.PeerSelection.Governor.KnownPeers`.
- **Tracer Forwarder backend routing**: `backends_for()` in `tracer.rs` now correctly returns `Some(TraceBackend::Forwarder)` for the `"Forwarder"` backend string. Previously mapped to `None`, silently dropping events despite the handler being fully implemented.
- **Mempool UTxO conflict detection**: Sync-pipeline mempool eviction now extracts consumed inputs from applied blocks (`extract_consumed_inputs()` in `sync.rs`) and calls `Mempool::remove_conflicting_inputs()` to evict double-spend-conflicting mempool transactions. Complements existing `remove_confirmed()` + `purge_expired()` eviction.
- **NtC handshake accept**: `ntc_accept()` in `ntc_peer.rs` implements full Node-to-Client handshake negotiation: decodes `ProposeVersions`, negotiates best common version from `NTC_SUPPORTED_VERSIONS` (V16–V9), validates network magic, returns `AcceptVersion` or `RefuseVersionMismatch`. Wired into `local_server.rs` replacing raw mux start. Reference: `Ouroboros.Network.Protocol.Handshake`.
- **TxSubmission server blocking-first**: Inbound `TxSubmissionServer` uses always-blocking `request_tx_ids(true, ack, batch)` in its main collection loop matching upstream `txSubmissionServer` basic pattern. Reference: `Ouroboros.Network.TxSubmission.Inbound`.
- **Reconnect peer punishment**: All three reconnecting sync functions now demote offending peers to `PeerStatus::PeerCold` on `ReconnectAndPunish` disposition so governor backoff/forget logic penalizes misbehaving peers. Reference: upstream `InvalidBlockPunishment` closes connection.
- **Reconnect exponential backoff**: `ReconnectingRunState` tracks `consecutive_failures` (reset on successful batch). `reconnect_backoff()` applies exponential delay (1s base, 60s cap) from the second consecutive failure. First reconnection (peer rotation) proceeds immediately. Wired into all three reconnecting sync outer loops with shutdown-aware `tokio::select`. Reference: `peerBackerOff` in `Ouroboros.Network.PeerSelection.Governor`.
