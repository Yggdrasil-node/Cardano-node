Focus on runtime composition of network clients and orchestration helpers that remain thin integration layers.

## Scope
- `main.rs` (CLI entry point), `config.rs` (JSON config types), `runtime.rs`, `sync.rs`, and library exports under `node/src`.
- Peer bootstrap wiring, configuration parsing, and sync control flow coordination.

##  Rules *Non-Negotiable*
- Keep ledger and consensus business rules outside `node/src`.
- Keep reusable peer-policy and protocol-adjacent state outside `node/src`; if topology logic stops being simple config resolution and starts maintaining policy or state, it belongs in a crate.
- Favor small, explicit orchestration steps that are easy to smoke-test.
- Propagate typed errors and avoid hidden retries/backoff logic unless explicitly required.
- Keep naming close to `cardano-node` operational concepts.
- Public orchestration helpers MUST include Rustdocs when flow is non-trivial.
- Stay true to the official type naming and terminology for node concepts, network protocols, and ledger types when possible.
- Always read the folder specific `**/AGENTS.md` files. They MUST stay current and MUST remain operational rather than long-form documentation. If the folder context is outdated, missing, or incorrect, update the relevant AGENTS.md file.

## Current Phase
- **CLI**: `main.rs` uses `clap` with `run` (connect + sync), `validate-config` (operator preflight for config, peer-snapshot inputs, and any existing storage recovery state), `status` (inspect on-disk storage and report current sync position, block counts, and checkpoint state), and `default-config` (emit JSON) subcommands. CLI flags (`--peer`, `--network-magic`, `--no-verify`, `--batch-size`, `--checkpoint-interval-slots`, `--max-ledger-snapshots`, `--checkpoint-trace-max-frequency`, `--checkpoint-trace-severity`, `--checkpoint-trace-backend`, `--metrics-port`, `--config`, `--network`) override config-file values on `run`. `--network` accepts `mainnet`, `preprod`, or `preview` presets. When `--peer` is omitted, `run` tries the configured primary relay and then topology-derived fallback peers. `--metrics-port` enables a lightweight HTTP endpoint on `127.0.0.1:<port>` serving `GET /metrics` (Prometheus text exposition), `GET /metrics/json` (JSON snapshot), and `GET /health` (JSON health check with `status`, `uptime_seconds`, `blocks_synced`, `current_slot` for orchestrator liveness probes).
- **Config**: `config.rs` defines `NodeConfigFile` (serde JSON) with primary `peer_addr`, ordered `bootstrap_peers`, richer topology metadata (`local_roots`, `public_roots`, `use_ledger_after_slot`, `peer_snapshot_file`), durable `storage_dir`, network magic, protocol versions, KES params, keepalive interval, and upstream-style tracing fields (`TurnOnLogging`, `UseTraceDispatcher`, `TurnOnLogMetrics`, `TraceOptionNodeName`, `TraceOptions`, `TraceOptionForwarder`, `TraceOptionMetricsPrefix`, `TraceOptionResourceFrequency`). `default_config()` returns mainnet defaults. `NetworkPreset` enum with `to_config()` provides per-network constructors, and preset peer ordering is parsed from the vendored `node/configuration/*/topology.json` files with bootstrap peers preferred before trustable local roots, other local roots, and public roots. The topology domain types and topology-file root configuration itself now come from `yggdrasil-network`, including local-root semantics such as `hotValency`, `warmValency`, `diffusionMode`, `UseBootstrapPeers`, and `UseLedgerPeers`. `NodeConfigFile` also exposes `topology_config()`, `use_ledger_peers_policy()`, `parse_peer_snapshot_json()`/`load_peer_snapshot_file()`, and `eligible_ledger_fallback_peers()` so runtime code can consume network-owned ledger-peer policy without re-implementing `useLedgerAfterSlot`, latest-slot, ledger-state, or peer-snapshot freshness gating locally. `main.rs` now recovers ledger state at startup, extracts directly dialable registered pool relays from immutable ledger state, and merges those with any snapshot peers before evaluating the same network-owned policy gate. `runtime.rs` now also refreshes ChainDb reconnect fallback peers from the current checkpoint-tracked ledger pool relays and resolved snapshot file before each bootstrap attempt when a `UseLedgerPeers` policy is supplied.
- Boundary rule: `config.rs` may parse the upstream node JSON shape and hand operational values into other crates, including `peerSnapshotFile` JSON decoding into normalized peer sets, but candidate sequencing, reconnect attempt ordering, preferred-peer retry state, topology-group semantics (`hotValency`, `warmValency`, `diffusionMode`), peer governor behavior, ledger peers, peer sharing, or any long-lived peer registry should be implemented in `crates/network` once introduced.
- Upstream planning rule: keep following the official split where the consensus side exposes latest-slot, ledger-state judgement, and immutable-ledger peer data, while the network side owns root-set providers, peer-source tracking, and peer-selection state.
- Bootstrap wiring is implemented (`NodeConfig`, `PeerSession`, `bootstrap`, `bootstrap_with_fallbacks`).
- Full sync orchestration stack: `sync_step`, typed decode bridges, bounded loops, intersection finding, batch apply, managed sync service with `tokio::select!` shutdown + `ctrl_c` signal handling. Typed sync now consumes decoded ChainSync headers, typed point/tip payloads, decoded Shelley blocks, and multi-era BlockFetch batch helpers from `yggdrasil-network`. Verified sync also uses raw+decoded BlockFetch batch helpers while keeping body-hash checks and verification policy in `node`. `run_reconnecting_verified_sync_service` sits above that layer and only retries bootstrap, ChainSync, and BlockFetch connectivity failures.
- `tracer.rs` now provides `NodeTracer` and `trace_fields`, a thin stdout trace dispatcher that reads `TraceOptions`, enforces `maxFrequency` per namespace, and emits upstream-style namespace/severity trace objects in human or machine format. `NodeMetrics` exposes atomic operational counters (blocks synced, rollbacks, batches, stable blocks promoted, reconnects, current slot, block number, checkpoint slot) with a lock-free `MetricsSnapshot` for status queries and the Prometheus `/metrics` endpoint. `MetricsSnapshot::to_prometheus_text()` renders counters and gauges in Prometheus text exposition format. `NodeMetrics` is wired into the hot sync loops via `record_verified_batch_progress` and the coordinated-storage ChainDB path, updating counters for blocks synced, rollbacks, batches completed, stable blocks promoted, reconnects, current slot, and checkpoint slot. `serve_metrics` in `main.rs` is a raw tokio TCP handler responding to `GET /metrics` and `GET /metrics/json` on the port selected by `--metrics-port`, requiring no HTTP framework dependency. `run_reconnecting_verified_sync_service_with_tracer` adds traced bootstrap attempts, reconnects, batch completion, shutdown/failure reporting, and checkpoint cadence events while preserving the untraced wrapper for callers that do not want local trace output. Default config now carries a dedicated `Node.Recovery.Checkpoint` namespace override with `maxFrequency: 1.0` so checkpoint events can be routed or silenced without mutating the root trace policy.
- Multi-era block decode for all 7 era tags (Byron through Conway). Byron blocks are structurally decoded via `ByronBlock::decode_ebb()`/`decode_main()`, carrying epoch, slot, chain_difficulty, prev_hash, and raw header bytes. Alonzo (tag 5) uses dedicated `AlonzoBlock` (5-element format with `invalid_transactions` and TPraos header), distinct from the 4-element `ShelleyBlock` used for Shelley/Allegra/Mary (tags 2–4).
- Consensus header verification bridge for both Shelley-era (`shelley_header_to_consensus`, `verify_shelley_header`) and Praos-era (`praos_header_to_consensus`, `verify_praos_header`) headers. `verify_multi_era_block` dispatches to the correct verifier per era (Shelley through Alonzo → Shelley, Babbage/Conway → Praos).
- VRF data flow: bridge functions now carry leader VRF proof/output (and nonce VRF for TPraos) through to the consensus `HeaderBody`. `verify_block_vrf` extracts VRF key + leader proof from each era's block header and delegates to `verify_leader_proof`. `VrfVerificationParams` bundles epoch nonce, sigma, and active slot coefficient.
- Nonce evolution wiring: `apply_nonce_evolution` extracts the per-era VRF nonce contribution (TPraos `nonce_vrf` or Praos `vrf_result`) and prev_hash from a `MultiEraBlock` and feeds them to `NonceEvolutionState::apply_block`. Byron blocks are skipped. This enables epoch nonce tracking during sync without modifying `sync_batch_apply_verified`.
- Block body hash verification: `verify_block_body_hash` computes Blake2b-256 of body elements and compares against the hash declared in the header. `extract_header_block_body_hash` handles both 14-element (Praos) and 15-element (Shelley) header bodies. Wired into `sync_batch_apply_verified` via `VerificationConfig.verify_body_hash`.
- Block header hash computation uses real Blake2b-256.
- Mempool sync eviction: `extract_tx_ids` + `evict_confirmed_from_mempool`. TxSubmission runtime wiring now includes `serve_txsubmission_request_from_mempool`, a thin one-shot helper that answers a single `TxSubmissionClient` request from fee-ordered mempool state, plus `serve_txsubmission_request_from_reader` and `run_txsubmission_service`, which use an upstream-aligned `TxSubmissionMempoolReader`/`MempoolSnapshot` flow with a monotonic `last_idx` cursor so managed outbound serving only advertises transactions newer than the last acknowledged snapshot position. `run_txsubmission_service_shared` mirrors that managed flow for `SharedMempool` so the long-lived service can observe concurrent inserts between requests. For inbound-style admission, `add_tx_to_mempool`/`add_tx_to_shared_mempool` stage `LedgerState.apply_submitted_tx()` before committing the mempool insert, and `add_txs_to_mempool`/`add_txs_to_shared_mempool` repeat that logic in order so later transactions in a batch can depend on earlier accepted outputs while returning an explicit `MempoolAddTxResult` per transaction.
- Verified sync service: `run_verified_sync_service` with `VerifiedSyncServiceConfig` (batch_size + verification + optional nonce config + optional security_param + `LedgerCheckpointPolicy`) and `VerifiedSyncServiceOutcome` (includes final `NonceEvolutionState`, optional `ChainState`, and `stable_block_count`). Verified execution is split into fetch-only `sync_batch_verified` plus a ChainDb apply phase so coordinated storage mutation can happen after network I/O. `run_verified_sync_service_chaindb` now provides the same verified batch flow over coordinated storage so stable entries can be promoted through `yggdrasil-storage::ChainDb`; node still decides checkpoint cadence via `LedgerCheckpointPolicy`, but the underlying checkpoint clear, truncate, save, and retain mechanics now go through crate-owned `ChainDb` helpers. The traced reconnecting ChainDb runtime path emits `Node.Recovery.Checkpoint` events describing whether a checkpoint was persisted, skipped by cadence, or cleared, along with retention counts. Runtime also exposes `run_reconnecting_verified_sync_service_chaindb`, `run_reconnecting_verified_sync_service_chaindb_with_tracer`, `resume_reconnecting_verified_sync_service_chaindb`, and shared-ChainDb resume variants; CLI `run` now opens file-backed immutable, volatile, and ledger stores under `storage_dir`, can optionally bind `inbound_listen_addr`, and starts through the coordinated-storage resume path by default.
- Low-level coordinated-sync API note: `run_verified_sync_service_chaindb` now requires an explicit `base_ledger_state` argument instead of implicitly using `LedgerState::new(Era::Byron)`. Callers that want genesis-derived protocol parameters in replay validation must pass a genesis-seeded base state (for CLI/runtime this comes from `base_ledger_state(file_cfg, config_base_dir)` in `main.rs`).
- Recovery now includes `recover_ledger_state_chaindb`, a thin node-facing wrapper over storage-owned `ChainDb::recover_ledger_state()`. The storage crate owns typed checkpoint restore plus immutable/volatile suffix replay, while runtime still owns checkpoint cadence and reconnect orchestration. Both `run_reconnecting_verified_sync_service_chaindb` and `resume_reconnecting_verified_sync_service_chaindb` refresh the checkpoint store after successful verified batches.
- Runtime request helpers: `ReconnectingVerifiedSyncRequest::new(...)` and `ResumeReconnectingVerifiedSyncRequest::new(...)` now provide default-initialized constructors for optional reconnect fields while still requiring explicit base ledger state. Prefer these constructors in tests/helpers to keep reconnect call sites concise and to avoid silently dropping required base-state wiring.
- Runtime request helper usage: `main.rs` now builds its shared-ChainDb resumed sync request via `ResumeReconnectingVerifiedSyncRequest::new(...)` and then sets only non-default fields (`nonce_state`, `use_ledger_peers`, `peer_snapshot_path`, `metrics`), mirroring test usage and reducing manual request boilerplate.
- Fluent setter usage: request helpers now expose `with_nonce_state(...)`, `with_use_ledger_peers(...)`, `with_peer_snapshot_path(...)`, `with_metrics(...)`, `with_peer_registry(...)`, and `with_mempool(...)` so call sites can compose optional fields via method chaining rather than mutable field assignment.
- Epoch boundary wiring: `advance_ledger_with_epoch_boundary()` in sync.rs detects epoch transitions during block application using `is_new_epoch()` / `slot_to_epoch()` and calls `apply_epoch_boundary()` before the first block of each new epoch. `LedgerCheckpointTracking` optionally carries `StakeSnapshots` + `EpochSize`; when present, `update_ledger_checkpoint_after_progress` uses epoch-aware advancement instead of plain `advance_ledger_state_with_progress`. Epoch boundary is automatically enabled when `nonce_config` provides `epoch_size`. `LedgerCheckpointTracking` also carries `pool_block_counts: BTreeMap<PoolKeyHash, u64>` to accumulate per-pool block production within each epoch; `advance_ledger_with_epoch_boundary` hashes each block's issuer VKey (Blake2b-224) to a `PoolKeyHash` and increments the count, then at epoch transitions calls `compute_pool_performance()` to derive actual `UnitInterval` performance ratios from the `set` stake snapshot before passing them to `apply_epoch_boundary()`. Counts are cleared after each epoch boundary and on rollback. Both `advance_ledger_state_with_progress` and `advance_ledger_with_epoch_boundary` accept `evaluator: Option<&dyn PlutusEvaluator>` and call `apply_block_validated()`. Checkpoint-tracked replay now carries a stored `CekPlutusEvaluator` inside `LedgerCheckpointTracking`, seeded from `VerifiedSyncServiceConfig.plutus_cost_model` when available, so rollback replay and epoch-boundary advancement use the same genesis-derived CEK costs instead of recreating `CekPlutusEvaluator::new()` for each batch.
- **Plutus script evaluation** (`plutus_eval.rs`): `CekPlutusEvaluator` implements `yggdrasil_ledger::plutus_validation::PlutusEvaluator` using the `yggdrasil-plutus` CEK machine. It decodes script bytes via `decode_script_bytes`, applies `datum` (if spending), `redeemer`, and a version-aware `ScriptContext` derived from the normalized ledger `TxContext`, evaluates within the `ExUnits` budget declared by the transaction, and returns `Ok(())` for V1/V2 non-error outcomes or requires `Bool(true)` for V3. `TxInfo` now includes resolved inputs/reference inputs, outputs with structured Shelley-family addresses, fee, mint, withdrawals (V2 staking-credential keyed, V3 credential keyed), certificates, signatories, redeemers, datums, tx id, Conway votes/proposals, and treasury fields. Byron/reward addresses are omitted from TxOut translation rather than encoded as bogus raw bytes. The V3 certifying path translates the supported V3 `TxCert` forms we can derive from the typed ledger certificate set and now fails explicitly on unsupported certificates such as `GenesisDelegation` instead of fabricating integer placeholders; the proposing path likewise fails explicitly if a `ProposalProcedure` cannot be translated (for example malformed reward-account bytes). On the legacy V1/V2 path, deposit-bearing staking certificates now translate as legacy reg/de-reg certs, while unsupported Conway-only certificates and Conway governance/treasury fields fail explicitly instead of silently falling back or being ignored. `CekPlutusEvaluator::with_cost_model()` allows a custom cost model; node config now loads per-builtin parameterized CEK costing from named Alonzo genesis `costModels` maps via `NodeConfigFile::load_plutus_cost_model()`, and when those maps are absent it now converts the live 251-entry Conway `plutusV3CostModel` array into the same named/per-builtin pipeline before threading the calibrated model into checkpoint-tracked ledger replay. `Default` still falls back to `CostModel::default()` when no genesis cost model is configured.
- ChainState integration: `multi_era_block_to_chain_entry` extracts `ChainEntry` from multi-era blocks (all eras including Byron), `track_chain_state_entries` exposes the drained stable prefix, `track_chain_state` preserves the count-returning wrapper, `promote_stable_blocks` copies stable blocks from volatile to immutable store, and coordinated-storage paths now convert the last stable `ChainEntry` to a `Point` and call `ChainDb::promote_volatile_prefix()` directly.
- Genesis parameters: `NodeConfigFile` now has `epoch_length`, `security_param_k`, `active_slot_coeff` (all with serde defaults). `run` command computes `stability_window = 3k/f` and builds `NonceEvolutionConfig`.
- Genesis parameters: `NodeConfigFile` now has `epoch_length`, `security_param_k`, `active_slot_coeff` (all with serde defaults). `run` command computes `stability_window = 3k/f` and builds `NonceEvolutionConfig`. **Phase 7**: `NodeConfigFile` now also exposes `ShelleyGenesisFile`, `AlonzoGenesisFile`, and `ConwayGenesisFile` optional fields (matching official Cardano node config keys). `load_genesis_protocol_params(&self, config_base_dir) -> Result<Option<ProtocolParameters>>` resolves these relative paths, parses the JSON via `genesis.rs` serde types, and calls `build_protocol_parameters(shelley, alonzo, conway)` to assemble network-accurate validation thresholds. Preset configs (`mainnet_config()`, `preprod_config()`, `preview_config()`) point to the vendored genesis files under `node/configuration/*/`. `main.rs` now centralizes this in a `base_ledger_state(file_cfg, config_base_dir)` helper and uses that genesis-seeded state consistently for startup peer-selection recovery, `validate-config`, `status`, and the resumed sync service, so all recovery/reporting paths replay blocks against the same protocol-parameter baseline rather than only the live sync path.
- **`genesis.rs`**: New module providing `ShelleyGenesis`, `AlonzoGenesis`, `ConwayGenesis` serde types plus `GenesisRational` (handles both float and `{numerator,denominator}` JSON forms) for reading era genesis files. `build_protocol_parameters(shelley, alonzo, conway)` assembles `ProtocolParameters` mapping Shelley fee/staking/KES fields, Shelley `protocolParams.protocolVersion`, and Alonzo execution-price/collateral fields. `ShelleyGenesis` now also parses upstream `networkId`, `networkMagic`, `genDelegs`, `initialFunds`, and `staking`; `build_shelley_genesis_bootstrap()` converts that into a typed bootstrap bundle and derives Shelley initial-funds pseudo `TxIn`s by hashing the serialized address bytes, matching upstream `initialFundsPseudoTxIn` structure, while also parsing static genesis stake delegations for first-Shelley activation. `load_shelley_genesis`, `load_shelley_genesis_bootstrap`, `load_alonzo_genesis`, and `load_conway_genesis` are simple `serde_json` file loaders returning `GenesisLoadError`. `main.rs` keeps the base ledger state rooted in Byron but now attaches the parsed Shelley bootstrap bundle as pending ledger state so recovery and replay activate initial funds and static stake delegations only once the first Shelley-family block is applied.
- **Inbound server session** (`server.rs`): `InboundPeerSession` wraps the five server drivers for an accepted inbound peer. `run_keepalive_server` runs the KeepAlive echo loop. `run_blockfetch_server` and `run_chainsync_server` serve blocks/headers via `BlockProvider`/`ChainProvider` traits. `run_txsubmission_server` now drives the TxSubmission2 server side by requesting transaction ids and bodies from the remote client and forwarding those bodies into a `TxSubmissionConsumer`. `run_peersharing_server` drives the PeerSharing server side via a `PeerSharingProvider` trait; `SharedPeerSharingProvider` implements this over a shared `PeerRegistry`, returning only warm/hot peers up to the requested amount. `run_inbound_accept_loop` accepts inbound connections via `PeerListener`, spawning KeepAlive plus optional BlockFetch, ChainSync, TxSubmission, and PeerSharing server tasks when providers/consumers/peer-sharing-provider are supplied. `SharedChainDb<I,V,L>` wraps `Arc<RwLock<ChainDb>>` (following `SharedMempool` pattern) and implements both `BlockProvider` and `ChainProvider` over coordinated immutable+volatile storage with CBOR point decode, `(from, to]` BlockFetch range semantics, and ChainSync cursor advancement keyed by encoded `Point`. ChainSync roll-forward payloads are header bytes re-derived from stored `Block.raw_cbor`, not full block payloads. `Block.raw_cbor` carries original wire bytes through the sync pipeline so BlockFetch can re-serve them. `MultiEraSyncStep::RollForward` now carries `raw_blocks: Option<Vec<Vec<u8>>>`; verified fetch preserves those bytes before the apply phase stores them. `SharedTxSubmissionConsumer` recovers the current ledger snapshot from coordinated storage, decodes submitted transactions against the current ledger era, and admits accepted ones into a shared mempool. `main.rs` now binds the inbound listener when `NodeConfigFile.inbound_listen_addr` is set and runs it against the same shared coordinated ChainDb used by sync.
- **Governor loop wiring** (`config.rs`, `runtime.rs`, `main.rs`): `NodeConfigFile` now carries governor interval and target counts, presets/default JSON include those fields, `RuntimeGovernorConfig` bridges config into runtime, and `main.rs` seeds a shared `PeerRegistry` from the primary peer plus topology-owned root sources before spawning `run_governor_loop()` alongside sync and inbound serving. `runtime.rs` now also owns an `OutboundPeerManager` that makes the loop effectful for warm peers: `PromoteToWarm` dials a dedicated `PeerSession`, warm peers are retained and heartbeat-driven with KeepAlive, `DemoteToCold` aborts the mux, and failed warm keepalives feed back into governor failure tracking. The same loop now refreshes root peers through `DnsRootPeerProvider` + `RootPeerProviderState` and reconciles ledger peers from shared `ChainDb` recovery plus optional `peerSnapshotFile` through the network crate policy helpers, using the caller-supplied genesis-seeded `base_ledger_state` rather than an implicit Byron default. Future work should attach `PromoteToHot` peers to real sync/data scheduling and connect the registry to PeerSharing.
- **Governor loop wiring** (`config.rs`, `runtime.rs`, `main.rs`): `NodeConfigFile` now carries governor interval and target counts, presets/default JSON include those fields, `RuntimeGovernorConfig` bridges config into runtime, and `main.rs` seeds a shared `PeerRegistry` from the primary peer plus topology-owned root sources before spawning `run_governor_loop()` alongside sync and inbound serving. `runtime.rs` now also owns an `OutboundPeerManager` that makes the loop effectful for warm peers: `PromoteToWarm` dials a dedicated `PeerSession`, warm peers are retained and heartbeat-driven with KeepAlive, `DemoteToCold` aborts the mux, and failed warm keepalives feed back into governor failure tracking. `PromoteToHot` now marks a warm peer as hot via `OutboundPeerManager::promote_to_hot()` (sets `is_hot` flag on `ManagedWarmPeer`), and `DemoteToWarm` reverses that via `demote_to_warm()`, enabling governor-driven hot/warm lifecycle without tearing down the underlying warm session. Warm sessions now also request shared peers over mini-protocol 10 when the handshake negotiated `peer_sharing > 0`, and the governor reconciles those discoveries into `PeerRegistry` as `PeerSourcePeerShare`. The same loop refreshes root peers through `DnsRootPeerProvider` + `RootPeerProviderState` and reconciles ledger peers from shared `ChainDb` recovery plus optional `peerSnapshotFile` through the network crate policy helpers, using the caller-supplied genesis-seeded `base_ledger_state` rather than an implicit Byron default. `main.rs` constructs a `SharedPeerSharingProvider` from the same shared `PeerRegistry` and passes it to the inbound accept loop so server-side PeerSharing responds with the node's known warm/hot peers. Hot peers now have their chain tips refreshed each governor tick via `refresh_hot_peer_tips()`, which queries each hot peer's ChainSync `find_intersect_points` at Origin; `best_hot_peer()` selects the hot peer with the highest tip slot for future chain-selection use. The governor loop now also takes an optional `SharedMempool` and calls `purge_expired(tip_slot)` each tick to evict TTL-expired transactions. `main.rs` wires a single `SharedMempool` into both the governor loop and the inbound TxSubmission consumer. `collect_rolled_back_tx_ids` in `sync.rs` captures transaction IDs from blocks that will be discarded during a rollback using `VolatileStore::suffix_after`, enabling future re-admission of rolled-back transactions. Future work should attach hot peers to real sync/data scheduling.
- **NtC local socket server** (`local_server.rs`): `NtcLocalServer` drives LocalTxSubmission, LocalStateQuery, and LocalTxMonitor mini-protocols over a Unix domain socket, mirroring the Haskell node's NtC interface. `BasicLocalQueryDispatcher` handles 8 query tags: (0) CurrentEra, (1) ChainTip, (2) CurrentEpoch, (3) ProtocolParameters, (4) UTxOByAddress (takes address bytes param), (5) StakeDistribution (pool state iteration), (6) RewardBalance (takes reward account bytes param), (7) TreasuryAndReserves ([treasury, reserves]). All queries operate via `LedgerStateSnapshot` and return opaque CBOR blobs. LocalTxMonitor is wired into `SharedMempool` for pending transaction visibility. LocalTxSubmission admission reuses the same staged `apply_submitted_tx` → shared mempool insert path as inbound NtN submission.
- **Genesis constitution wiring** (`genesis.rs`, `config.rs`, `main.rs`): `ConwayGenesis` now includes a `constitution` field (`GenesisConstitution`) parsing the `constitution.anchor` (url + dataHash) and `constitution.script` (guardrails script hash) from `conway-genesis.json`. `build_genesis_enact_state(conway)` constructs an `EnactState` with the genesis constitution and guardrails script hash. `NodeConfigFile::load_genesis_enact_state()` loads the Conway genesis and calls the builder. `strict_base_ledger_state()` in `main.rs` now applies the genesis enact state to the base ledger state so governance validation has the correct initial constitution at startup.
- **Hot-peer sync preference** (`runtime.rs`, `main.rs`): The reconnecting verified sync loop now queries the shared `PeerRegistry` for `preferred_hot_peer()` at each reconnect attempt, preferring the hot peer with the highest known tip slot (`hot_tip_slot`) when available. The governor updates those tip slots during `refresh_hot_peer_tips()` and clears stale values on non-hot status transitions. Reconnect bootstrap preference order is now explicit: `preferred_hot_peer()` first, then the previous successful peer as fallback. This gives both ChainDb reconnect paths (shared and non-shared) deterministic, tip-aware hot-peer bootstrap bias. The `ResumeReconnectingVerifiedSyncRequest` and `ReconnectingVerifiedSyncContext` carry `peer_registry: Option<Arc<RwLock<PeerRegistry>>>`, and `main.rs` wires the same shared registry used by the governor loop into the sync request via `.with_peer_registry()`.
- **Mempool eviction in sync loop** (`runtime.rs`, `main.rs`): The `ResumeReconnectingVerifiedSyncRequest` and `ReconnectingVerifiedSyncContext` now carry `mempool: Option<SharedMempool>` with a `.with_mempool()` fluent setter. Both reconnecting ChainDb inner sync functions iterate each `RollForward` step after a batch, extract confirmed tx IDs via `extract_tx_ids`, and call `mempool.remove_confirmed()` plus `mempool.purge_expired()`, tracing eviction counts under `Mempool.Eviction`. `main.rs` wires the same `SharedMempool` used by the governor loop and inbound TxSubmission into the sync request.
- **Rollback transaction collection** (`sync.rs`, `runtime.rs`): `apply_verified_progress_to_chaindb` now calls `collect_rolled_back_tx_ids` before each `RollBackward` step and returns the gathered tx IDs in `AppliedVerifiedProgress.rolled_back_tx_ids`. Both the non-shared and shared ChainDb inner functions trace rolled-back tx counts under `ChainDB.Rollback` when any transactions are collected during a batch.
- **VRF verification config** (`main.rs`): The sync config now wires `ActiveSlotCoeff` from the genesis `active_slot_coeff` config field and enables `verify_vrf` when the coefficient is valid. This completes the VRF verification data path so the runtime can check block VRF proofs against pool stake during sync.
