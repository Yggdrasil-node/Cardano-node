#  Guidance for node runtime and sync orchestration implementation details
Focus on runtime composition of network clients and orchestration helpers that remain thin integration layers.

## Scope
- `main.rs` (CLI entry point), `config.rs` (upstream-aligned config types loaded from JSON or YAML), `runtime.rs`, `sync.rs`, and library exports under `node/src`.
- Peer bootstrap wiring, configuration parsing, and sync control flow coordination.

##  Rules *Non-Negotiable*
- Keep ledger and consensus business rules outside `node/src`.
- Keep reusable peer-policy and protocol-adjacent state outside `node/src`; if topology logic stops being simple config resolution and starts maintaining policy or state, it belongs in a crate.
- Favor small, explicit orchestration steps that are easy to smoke-test.
- Propagate typed errors and avoid hidden retries/backoff logic unless explicitly required.
- Keep naming close to `cardano-node` operational concepts.
- Public orchestration helpers MUST include Rustdocs when flow is non-trivial.
- Stay true to the official type naming and terminology for node concepts, network protocols, and ledger types when possible.
- Always read the folder specific `**/AGENTS.md` files. They MUST stay current and MUST remain operational rather than long-form documentation. If the folder context is outdated, missing, or incorrect, update the relevant `AGENTS.md` file.

## Current Phase
- **Connection manager runtime integration (Step 1.2)**: `main.rs` now creates a shared `ConnectionManagerState` and passes it to both `run_governor_loop()` and `run_inbound_accept_loop()`. In `runtime.rs`, outbound connection lifecycle actions are now driven through CM transitions (`acquire_outbound_connection` + `outbound_handshake_done`, `release_outbound_connection`) instead of direct status mutation, and returned `CmAction`s are applied deterministically in runtime order.
- **Inbound CM tracking (Step 1.2)**: `server.rs` now optionally feeds accepted peers through `include_inbound_connection` + `inbound_handshake_done` and calls `release_inbound_connection` on per-peer teardown so inbound accept no longer bypasses CM state accounting.
- **Inbound governor runtime integration (Step 1.3)**: `main.rs` now creates a shared `InboundGovernorState` and threads it into inbound serving and peer sharing. `server.rs` now emits IG events for new connections, responder counter deltas, inactivity/maturation ticks, and mux-finished teardown; IG-produced actions are bridged through CM calls (`promoted_to_warm_remote`, `demoted_to_cold_remote`, `release_inbound_connection` with `apply_commit_result`). `SharedPeerSharingProvider` can now include mature IG duplex peers in responses.
- **Control-message wiring (Step 1.4)**: `runtime.rs` now stores `TemperatureBundle<ControlMessage>` per managed outbound peer and applies lifecycle transitions on establish/activate/deactivate/close using explicit `Continue`/`Quiesce`/`Terminate` control states.
- **Churn + judgement feedback wiring (Steps 1.5/1.6)**: governor runtime now computes `LedgerStateJudgement` per tick, derives `PeerSelectionMode` and `AssociationMode` via upstream helpers, and updates churn regime/fetch mode before `GovernorState::tick()`.
- **Peer metrics integration (Step 1.7)**: runtime now feeds `GovernorState.metrics` from hot-peer tip observations and selected best-hot-peer choices, and includes `combined_score` in governor action traces for parity debugging.
- **Connection-manager timeout + drain enforcement (Step 1.8)**: `run_governor_loop()` now runs `ConnectionManagerState::timeout_tick(Instant::now())` each cycle and applies resulting `CmAction`s via the existing CM action bridge, so responder and time-wait CM transitions are no longer purely latent state-machine functions. On shutdown, the governor performs a graceful outbound drain: Phase 1 signals all outbound peers `ControlMessage::Terminate` via `apply_control_close()`, Phase 2 releases CM-managed warm/hot peers through `release_outbound_connection`.
- **Inbound rate limiting**: `run_inbound_accept_loop()` now applies `rate_limit_decision()` with `AcceptedConnectionsLimit` (512 hard / 384 soft / 5s delay) after each `accept_peer()`. On `SoftDelay(d)` the loop sleeps before proceeding; on `HardLimit` the connection is aborted.
- **Inbound graceful shutdown**: `run_inbound_accept_loop()` now tracks spawned inbound sessions via `tokio::task::JoinSet` instead of fire-and-forget `tokio::spawn`. On shutdown: stops accepting, emits `CommitRemote` for tracked IG peers, waits up to 5s for tasks to complete, then force-aborts remaining tasks.
- **Dynamic hot-peer egress weights**: `PeerSession` now stores per-protocol `WeightHandle`s extracted during bootstrap. `promote_to_hot()` boosts ChainSync weight to 3 and BlockFetch weight to 2; `demote_to_warm()` resets all weights to 1. Changes take effect immediately in the mux writer's round-robin loop via shared `Arc<AtomicU8>`.
- **CLI**: `main.rs` uses `clap` with `run` (connect + sync), `validate-config` (operator preflight for config, peer-snapshot inputs, and any existing storage recovery state), `status` (inspect on-disk storage and report current sync position, block counts, and checkpoint state), and `default-config` (emit JSON) subcommands. CLI flags (`--peer`, `--network-magic`, `--no-verify`, `--batch-size`, `--checkpoint-interval-slots`, `--max-ledger-snapshots`, `--checkpoint-trace-max-frequency`, `--checkpoint-trace-severity`, `--checkpoint-trace-backend`, `--metrics-port`, `--config`, `--network`, `--topology`, `--database-path`, `--port`, `--host-addr`) override config-file values on `run`. `--network` accepts `mainnet`, `preprod`, or `preview` presets. `--topology` loads an external P2P topology file (upstream JSON format) and overrides inline topology config; `TopologyFilePath` config key provides the same capability from the config file. `--database-path` overrides `storage_dir` on `run`, `validate-config`, and `status`. `--port` and `--host-addr` override the inbound listen address on `run`. When `--peer` is omitted, `run` tries the configured primary relay and then topology-derived fallback peers. `--metrics-port` enables a lightweight HTTP endpoint on `127.0.0.1:<port>` serving `GET /metrics` (Prometheus text exposition), `GET /metrics/json` (JSON snapshot), and `GET /health` (JSON health check with `status`, `uptime_seconds`, `blocks_synced`, `current_slot` for orchestrator liveness probes).
- **Config**: `config.rs` defines `NodeConfigFile` (serde shape aligned to upstream config keys) with primary `peer_addr`, ordered `bootstrap_peers`, richer topology metadata (`local_roots`, `public_roots`, `use_ledger_after_slot`, `peer_snapshot_file`), durable `storage_dir`, network magic, protocol versions, KES params, keepalive interval, and upstream-style tracing fields (`TurnOnLogging`, `UseTraceDispatcher`, `TurnOnLogMetrics`, `TraceOptionNodeName`, `TraceOptions`, `TraceOptionForwarder`, `TraceOptionMetricsPrefix`, `TraceOptionResourceFrequency`). `main.rs` config loading accepts JSON first with YAML fallback for this schema. `default_config()` returns mainnet defaults. `NetworkPreset` enum with `to_config()` provides per-network constructors, and preset peer ordering is parsed from the vendored `node/configuration/*/topology.json` files with bootstrap peers preferred before trustable local roots, other local roots, and public roots. The topology domain types and topology-file root configuration itself now come from `yggdrasil-network`, including local-root semantics such as `hotValency`, `warmValency`, `diffusionMode`, `UseBootstrapPeers`, and `UseLedgerPeers`. `NodeConfigFile` also exposes `topology_config()`, `use_ledger_peers_policy()`, `parse_peer_snapshot_json()`/`load_peer_snapshot_file()`, and `eligible_ledger_fallback_peers()` so runtime code can consume network-owned ledger-peer policy without re-implementing `useLedgerAfterSlot`, latest-slot, ledger-state, or peer-snapshot freshness gating locally. `main.rs` now recovers ledger state at startup, extracts directly dialable registered pool relays from immutable ledger state, and merges those with any snapshot peers before evaluating the same network-owned policy gate. `runtime.rs` now also refreshes ChainDb reconnect fallback peers from the current checkpoint-tracked ledger pool relays and resolved snapshot file before each bootstrap attempt when a `UseLedgerPeers` policy is supplied.
- Boundary rule: `config.rs` may parse the upstream node JSON shape and hand operational values into other crates, including `peerSnapshotFile` JSON decoding into normalized peer sets, but candidate sequencing, reconnect attempt ordering, preferred-peer retry state, topology-group semantics (`hotValency`, `warmValency`, `diffusionMode`), peer governor behavior, ledger peers, peer sharing, or any long-lived peer registry should be implemented in `crates/network` once introduced.
- Upstream planning rule: keep following the official split where the consensus side exposes latest-slot, ledger-state judgement, and immutable-ledger peer data, while the network side owns root-set providers, peer-source tracking, and peer-selection state.

- Epoch boundary wiring: `advance_ledger_with_epoch_boundary()` in sync.rs detects epoch transitions during block application using `is_new_epoch()` / `slot_to_epoch()` and calls `apply_epoch_boundary()` before the first block of each new epoch. `LedgerCheckpointTracking` optionally carries `StakeSnapshots` + `EpochSize`; when present, `update_ledger_checkpoint_after_progress` uses epoch-aware advancement instead of plain `advance_ledger_state_with_progress`. Epoch boundary is automatically enabled when `nonce_config` provides `epoch_size`. `LedgerCheckpointTracking` also carries `pool_block_counts: BTreeMap<PoolKeyHash, u64>` to accumulate per-pool block production within each epoch; `advance_ledger_with_epoch_boundary` hashes each block's issuer VKey (Blake2b-224) to a `PoolKeyHash` and increments the count, then at epoch transitions calls `compute_pool_performance()` to derive actual `UnitInterval` performance ratios from the `set` stake snapshot before passing them to `apply_epoch_boundary()`. Counts are cleared after each epoch boundary and on rollback. Both `advance_ledger_state_with_progress` and `advance_ledger_with_epoch_boundary` accept `evaluator: Option<&dyn PlutusEvaluator>` and call `apply_block_validated()`. Checkpoint-tracked replay now carries a stored `CekPlutusEvaluator` inside `LedgerCheckpointTracking`, seeded from `VerifiedSyncServiceConfig.plutus_cost_model` when available, so rollback replay and epoch-boundary advancement use the same genesis-derived CEK costs instead of recreating `CekPlutusEvaluator::new()` for each batch.
- **Plutus script evaluation** (`plutus_eval.rs`): `CekPlutusEvaluator` implements `yggdrasil_ledger::plutus_validation::PlutusEvaluator` using the `yggdrasil-plutus` CEK machine. It decodes script bytes via `decode_script_bytes`, applies `datum` (if spending), `redeemer`, and a version-aware `ScriptContext` derived from the normalized ledger `TxContext`, evaluates within the `ExUnits` budget declared by the transaction, and returns `Ok(())` for V1/V2 non-error outcomes or requires `Bool(true)` for V3. `TxInfo` now includes resolved inputs/reference inputs, outputs with structured Shelley-family addresses, fee, mint, withdrawals (V2 staking-credential keyed, V3 credential keyed), certificates, signatories, redeemers, datums, tx id, Conway votes/proposals, and treasury fields. Byron/reward addresses are omitted from TxOut translation rather than encoded as bogus raw bytes. The V3 certifying path translates the supported V3 `TxCert` forms we can derive from the typed ledger certificate set and now fails explicitly on unsupported certificates such as `GenesisDelegation` instead of fabricating integer placeholders; the proposing path likewise fails explicitly if a `ProposalProcedure` cannot be translated (for example malformed reward-account bytes). On the legacy V1/V2 path, deposit-bearing staking certificates now translate as legacy reg/de-reg certs, while unsupported Conway-only certificates and Conway governance/treasury fields fail explicitly instead of silently falling back or being ignored. `CekPlutusEvaluator::with_cost_model()` allows a custom cost model; node config now loads per-builtin parameterized CEK costing from named Alonzo genesis `costModels` maps via `NodeConfigFile::load_plutus_cost_model()`, and when those maps are absent it now converts the live 251-entry Conway `plutusV3CostModel` array into the same named/per-builtin pipeline before threading the calibrated model into checkpoint-tracked ledger replay. `Default` still falls back to `CostModel::default()` when no genesis cost model is configured.

- **Inbound server session** (`server.rs`): `InboundPeerSession` wraps the five server drivers for an accepted inbound peer. `run_keepalive_server` runs the KeepAlive echo loop. `run_blockfetch_server` and `run_chainsync_server` serve blocks/headers via `BlockProvider`/`ChainProvider` traits. `run_txsubmission_server` now drives the TxSubmission2 server side by requesting transaction ids and bodies from the remote client and forwarding those bodies into a `TxSubmissionConsumer`. `run_peersharing_server` drives the PeerSharing server side via a `PeerSharingProvider` trait; `SharedPeerSharingProvider` implements this over a shared `PeerRegistry`, returning only warm/hot peers up to the requested amount. `run_inbound_accept_loop` accepts inbound connections via `PeerListener`, spawning KeepAlive plus optional BlockFetch, ChainSync, TxSubmission, and PeerSharing server tasks when providers/consumers/peer-sharing-provider are supplied. `SharedChainDb<I,V,L>` wraps `Arc<RwLock<ChainDb>>` (following `SharedMempool` pattern) and implements both `BlockProvider` and `ChainProvider` over coordinated immutable+volatile storage with CBOR point decode, `(from, to]` BlockFetch range semantics, and ChainSync cursor advancement keyed by encoded `Point`. ChainSync roll-forward payloads are header bytes re-derived from stored `Block.raw_cbor`, not full block payloads. `Block.raw_cbor` carries original wire bytes through the sync pipeline so BlockFetch can re-serve them. `MultiEraSyncStep::RollForward` now carries `raw_blocks: Option<Vec<Vec<u8>>>`; verified fetch preserves those bytes before the apply phase stores them. `SharedTxSubmissionConsumer` recovers the current ledger snapshot from coordinated storage, decodes submitted transactions against the current ledger era, and admits accepted ones into a shared mempool. `main.rs` now binds the inbound listener when `NodeConfigFile.inbound_listen_addr` is set and runs it against the same shared coordinated ChainDb used by sync.
- **Governor loop wiring** (`config.rs`, `runtime.rs`, `main.rs`): `NodeConfigFile` now carries governor interval and target counts, presets/default JSON include those fields, `RuntimeGovernorConfig` bridges config into runtime, and `main.rs` seeds a shared `PeerRegistry` from the primary peer plus topology-owned root sources before spawning `run_governor_loop()` alongside sync and inbound serving. `runtime.rs` now also owns an `OutboundPeerManager` that makes the loop effectful for warm peers: `PromoteToWarm` dials a dedicated `PeerSession`, warm peers are retained and heartbeat-driven with KeepAlive, `DemoteToCold` aborts the mux, and failed warm keepalives feed back into governor failure tracking.
