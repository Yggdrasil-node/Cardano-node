#  Guidance for node runtime and sync orchestration implementation details
Focus on runtime composition of network clients and orchestration helpers that remain thin integration layers.

## Scope
- `main.rs` (CLI entry point), `config.rs` (upstream-aligned config types loaded from JSON or YAML), `runtime.rs`, `sync.rs`, and library exports under `node/src`.
- Peer bootstrap wiring, configuration parsing, and sync control flow coordination.

##  Rules *Non-Negotiable*
- Keep ledger and consensus business rules outside `node/src`.
- Keep reusable peer-policy and protocol-adjacent state outside `node/src`; if topology logic stops being simple config resolution and starts maintaining policy or state, it belongs in a crate.
- Favor small, explicit orchestration steps that are easy to smoke-test.
- Propagate typed errors and avoid hidden retries/backoff logic unless explicitly required.
- Keep naming close to `cardano-node` operational concepts.
- Public orchestration helpers MUST include Rustdocs when flow is non-trivial.
- Stay true to the official type naming and terminology for node concepts, network protocols, and ledger types when possible.
- Always read the folder specific `**/AGENTS.md` files. They MUST stay current and MUST remain operational rather than long-form documentation. If the folder context is outdated, missing, or incorrect, update the relevant `AGENTS.md` file.

## Current Phase
- **Connection manager runtime integration (Step 1.2)**: `main.rs` now creates a shared `ConnectionManagerState` and passes it to both `run_governor_loop()` and `run_inbound_accept_loop()`. In `runtime.rs`, outbound connection lifecycle actions are now driven through CM transitions (`acquire_outbound_connection` + `outbound_handshake_done`, `release_outbound_connection`) instead of direct status mutation, and returned `CmAction`s are applied deterministically in runtime order.
- **Inbound CM tracking (Step 1.2)**: `server.rs` now optionally feeds accepted peers through `include_inbound_connection` + `inbound_handshake_done` and calls `release_inbound_connection` on per-peer teardown so inbound accept no longer bypasses CM state accounting.
- **Inbound governor runtime integration (Step 1.3)**: `main.rs` now creates a shared `InboundGovernorState` and threads it into inbound serving and peer sharing. `server.rs` now emits IG events for new connections, responder counter deltas, inactivity/maturation ticks, and mux-finished teardown; IG-produced actions are bridged through CM calls (`promoted_to_warm_remote`, `demoted_to_cold_remote`, `release_inbound_connection` with `apply_commit_result`). `SharedPeerSharingProvider` can now include mature IG duplex peers in responses.
- **Control-message wiring (Step 1.4)**: `runtime.rs` now stores `TemperatureBundle<ControlMessage>` per managed outbound peer and applies lifecycle transitions on establish/activate/deactivate/close using explicit `Continue`/`Quiesce`/`Terminate` control states.
- **Churn + judgement feedback wiring (Steps 1.5/1.6)**: governor runtime now computes `LedgerStateJudgement` per tick, derives `PeerSelectionMode` and `AssociationMode` via upstream helpers, and updates churn regime/fetch mode before `GovernorState::tick()`.
- **Config-driven peer-sharing/consensus mode wiring parity**: `NodeConfigFile` now exposes `peer_sharing` (wire value, default `1`) and `consensus_mode` (`PraosMode`/`GenesisMode`, default `PraosMode`). `main.rs` threads both into `RuntimeGovernorConfig` and `NodeConfig`; `run_governor_loop()` now uses configured peer-sharing willingness for `compute_association_mode()` and configured consensus mode for `pick_churn_regime()`; bootstrap handshake proposals now advertise configured `peer_sharing` instead of a hardcoded `1`.
- **Peer metrics integration (Step 1.7)**: runtime now feeds `GovernorState.metrics` from hot-peer tip observations and selected best-hot-peer choices, and includes `combined_score` in governor action traces for parity debugging.
- **Connection-manager timeout + drain enforcement (Step 1.8)**: `run_governor_loop()` now runs `ConnectionManagerState::timeout_tick(Instant::now())` each cycle and applies resulting `CmAction`s via the existing CM action bridge, so responder and time-wait CM transitions are no longer purely latent state-machine functions. On shutdown, the governor performs a graceful outbound drain: Phase 1 signals all outbound peers `ControlMessage::Terminate` via `apply_control_close()`, Phase 2 releases CM-managed warm/hot peers through `release_outbound_connection`.
- **Inbound rate limiting**: `run_inbound_accept_loop()` now applies `rate_limit_decision()` with `AcceptedConnectionsLimit` (512 hard / 384 soft / 5s delay) after each `accept_peer()`. On `SoftDelay(d)` the loop sleeps before proceeding; on `HardLimit` the connection is aborted.
- **Inbound graceful shutdown**: `run_inbound_accept_loop()` now tracks spawned inbound sessions via `tokio::task::JoinSet` instead of fire-and-forget `tokio::spawn`. On shutdown: stops accepting, emits `CommitRemote` for tracked IG peers, waits up to 5s for tasks to complete, then force-aborts remaining tasks.
- **Dynamic hot-peer egress weights**: `PeerSession` now stores per-protocol `WeightHandle`s extracted during bootstrap. `promote_to_hot()` boosts ChainSync weight to 3 and BlockFetch weight to 2; `demote_to_warm()` resets all weights to 1. Changes take effect immediately in the mux writer's round-robin loop via shared `Arc<AtomicU8>`.
- **CLI**: `main.rs` uses `clap` with `run` (connect + sync), `validate-config` (operator preflight for config, peer-snapshot inputs, role resolution, producer credentials, and any existing storage recovery state), `status` (inspect on-disk storage and report current sync position, block counts, and checkpoint state), `default-config` (emit JSON), `cardano-cli` (pure-Rust command group for `version`, `show-upstream-config`, and `query-tip` with upstream preset-aware config resolution), and Unix-only `query`/`submit-tx` NtC subcommands. CLI flags (`--peer`, `--network-magic`, `--no-verify`, `--batch-size`, `--checkpoint-interval-slots`, `--max-ledger-snapshots`, `--checkpoint-trace-max-frequency`, `--checkpoint-trace-severity`, `--checkpoint-trace-backend`, `--metrics-port`, `--config`, `--network`, `--topology`, `--database-path`, `--port`, `--host-addr`, `--non-producing-node`, and Shelley producer credential flags) override config-file values on `run` where applicable; `validate-config` accepts the role/listener/credential flags so its JSON report reflects the exact run-mode preflight. Producer credentials are atomic: all four Shelley paths are required for forging, partial sets fail unless `--non-producing-node` is set. `query` and `submit-tx` now connect through `yggdrasil-network::ntc_connect` (negotiated NtC handshake) and accept `--network-magic` (or `CARDANO_NODE_NETWORK_MAGIC`) for handshake parity. `--network` accepts `mainnet`, `preprod`, or `preview` presets. `--topology` loads an external P2P topology file (upstream JSON format) and overrides inline topology config; `TopologyFilePath` config key provides the same capability from the config file. `--database-path` overrides `storage_dir` on `run`, `validate-config`, and `status`. `--port` and `--host-addr` override the inbound listen address on `run` and `validate-config`. When `--peer` is omitted, `run` tries the configured primary relay and then topology-derived fallback peers. `--metrics-port` enables a lightweight HTTP endpoint on `127.0.0.1:<port>` serving `GET /metrics` (Prometheus text exposition), `GET /metrics/json` (JSON snapshot), and `GET /health` (JSON health check with `status`, `uptime_seconds`, `blocks_synced`, `current_slot` for orchestrator liveness probes), plus upstream-style debug aliases `GET /debug`, `GET /debug/metrics`, `GET /debug/metrics/prometheus`, and `GET /debug/health`.
- **Config**: `config.rs` defines `NodeConfigFile` (serde shape aligned to upstream config keys) with primary `peer_addr`, ordered `bootstrap_peers`, richer topology metadata (`local_roots`, `public_roots`, `use_ledger_after_slot`, `peer_snapshot_file`), durable `storage_dir`, network magic, protocol versions, KES params, keepalive interval, and upstream-style tracing fields (`TurnOnLogging`, `UseTraceDispatcher`, `TurnOnLogMetrics`, `TraceOptionNodeName`, `TraceOptions`, `TraceOptionForwarder`, `TraceOptionMetricsPrefix`, `TraceOptionResourceFrequency`). `main.rs` config loading accepts JSON first with YAML fallback for this schema. `default_config()` returns mainnet defaults. `NetworkPreset` enum with `to_config()` provides per-network constructors, and preset peer ordering is parsed from the vendored `node/configuration/*/topology.json` files with bootstrap peers preferred before trustable local roots, other local roots, and public roots. The topology domain types and topology-file root configuration itself now come from `yggdrasil-network`, including local-root semantics such as `hotValency`, `warmValency`, `diffusionMode`, `UseBootstrapPeers`, and `UseLedgerPeers`. `NodeConfigFile` also exposes `topology_config()`, `use_ledger_peers_policy()`, `parse_peer_snapshot_json()`/`load_peer_snapshot_file()`, and `eligible_ledger_fallback_peers()` so runtime code can consume network-owned ledger-peer policy without re-implementing `useLedgerAfterSlot`, latest-slot, ledger-state, or peer-snapshot freshness gating locally. `main.rs` now recovers ledger state at startup, extracts directly dialable registered pool relays from immutable ledger state, and merges those with any snapshot peers before evaluating the same network-owned policy gate. `runtime.rs` now also refreshes ChainDb reconnect fallback peers from the current checkpoint-tracked ledger pool relays and resolved snapshot file before each bootstrap attempt when a `UseLedgerPeers` policy is supplied.
- Boundary rule: `config.rs` may parse the upstream node JSON shape and hand operational values into other crates, including `peerSnapshotFile` JSON decoding into normalized peer sets, but candidate sequencing, reconnect attempt ordering, preferred-peer retry state, topology-group semantics (`hotValency`, `warmValency`, `diffusionMode`), peer governor behavior, ledger peers, peer sharing, or any long-lived peer registry should be implemented in `crates/network` once introduced.
- Upstream planning rule: keep following the official split where the consensus side exposes latest-slot, ledger-state judgement, and immutable-ledger peer data, while the network side owns root-set providers, peer-source tracking, and peer-selection state.
- Bootstrap wiring is implemented (`NodeConfig`, `PeerSession`, `bootstrap`, `bootstrap_with_fallbacks`).
- Full sync orchestration stack: `sync_step`, typed decode bridges, bounded loops, intersection finding, batch apply, managed sync service with `tokio::select!` shutdown + `ctrl_c` signal handling. Typed sync now consumes decoded ChainSync headers, typed point/tip payloads, decoded Shelley blocks, and multi-era BlockFetch batch helpers from `yggdrasil-network`. Verified sync also uses raw+decoded BlockFetch batch helpers while keeping body-hash checks and verification policy in `node`. When a verified batch starts from `Point::Origin` and collects multiple ChainSync roll-forward headers before fetching bodies, BlockFetch must use the first announced concrete header as the lower bound, not collapse the range to the final upper bound; otherwise the initial prefix is silently skipped and later Byron spends fail with UTxO holes. `run_reconnecting_verified_sync_service` sits above that layer and only retries bootstrap, ChainSync, and BlockFetch connectivity failures.
- **Diffusion pipelining (DPvDV) wiring**: `main.rs` now creates optional shared `TentativeState` (currently enabled via `DiffusionPipeliningSupport::DiffusionPipeliningOn`) and threads it through both inbound `SharedChainDb::from_arc_with_tentative()` and resumed reconnecting sync requests (`with_tentative_state`). `sync_batch_verified_with_tentative()` now sets tentative headers on roll-forward announcements using decoded Shelley/Praos header views and clears them as adopted on success or trap on validation/fetch errors. `run_chainsync_server()` serves tentative tips and rolls followers back to the confirmed tip when a served tentative header is trapped. Coverage: `server::tests::chainsync_server_rolls_back_after_tentative_trap` and `runtime_reconnecting_sync_traps_tentative_header_on_validation_failure`.
- `tracer.rs` now provides `NodeTracer`, `trace_fields`, and `TraceDetail`, a thin stdout trace dispatcher that reads `TraceOptions`, enforces severity hierarchy threshold filtering, uses longest-prefix namespace matching for severity/backends/`maxFrequency`/detail, and emits upstream-style namespace/severity trace objects in human (plain or ANSI-coloured), or machine (JSON) format. `TraceDetail` (DMinimal/DNormal/DDetailed/DMaximum) mirrors upstream `DetailLevel`; `detail_for()` resolves the per-namespace detail level and `trace_runtime_detailed()` gates events by minimum detail. All upstream backend strings (`EKGBackend`, `Forwarder`, `PrometheusSimple`, `Stdout HumanFormatColoured`, `Stdout HumanFormatUncoloured`, `Stdout MachineFormat`) are now recognised. `NodeTracer::clone()` now preserves the `Forwarder` transport and shares the same `maxFrequency` rate-limiter state (`last_emit_ms`) across clones so spawned runtime tasks do not bypass namespace throttle limits. `NodeMetrics` exposes atomic operational counters (blocks synced, rollbacks, batches, stable blocks promoted, reconnects, current slot, block number, checkpoint slot) with a lock-free `MetricsSnapshot` for status queries and the Prometheus `/metrics` endpoint. `MetricsSnapshot::to_prometheus_text()` renders counters and gauges in Prometheus text exposition format. `NodeMetrics` is wired into the hot sync loops via `record_verified_batch_progress` and the coordinated-storage ChainDB path, updating counters for blocks synced, rollbacks, batches completed, stable blocks promoted, reconnects, current slot, and checkpoint slot. `serve_metrics` in `main.rs` is a raw tokio TCP handler responding to `GET /metrics` and `GET /metrics/json` on the port selected by `--metrics-port`, requiring no HTTP framework dependency. `run_reconnecting_verified_sync_service_with_tracer` adds traced bootstrap attempts, reconnects, batch completion, shutdown/failure reporting, and checkpoint cadence events while preserving the untraced wrapper for callers that do not want local trace output. Default config now carries a dedicated `Node.Recovery.Checkpoint` namespace override with `maxFrequency: 1.0` so checkpoint events can be routed or silenced without mutating the root trace policy.
- Multi-era block decode for all 7 era tags (Byron through Conway). Byron blocks are structurally decoded via `ByronBlock::decode_ebb()`/`decode_main()`, carrying epoch, slot, chain_difficulty, prev_hash, and raw header bytes. Alonzo (tag 5) uses dedicated `AlonzoBlock` (5-element format with `invalid_transactions` and TPraos header), distinct from the 4-element `ShelleyBlock` used for Shelley/Allegra/Mary (tags 2–4).
- Consensus header verification bridge for both Shelley-era (`shelley_header_to_consensus`, `verify_shelley_header`) and Praos-era (`praos_header_to_consensus`, `verify_praos_header`) headers. `verify_multi_era_block` dispatches to the correct verifier per era (Shelley through Alonzo → Shelley, Babbage/Conway → Praos). Additionally performs BBODY-level `validate_block_protocol_version` check (era/protocol-version consistency, reference: hard-fork combinator era transitions) before header verification. `validate_block_body_size` checks declared vs actual serialized body size (reference: `WrongBlockBodySizeBBODY` in `Cardano.Ledger.Shelley.Rules.Bbody`). Both functions are peer-attributable errors that trigger reconnection.
- VRF data flow: bridge functions now carry leader VRF proof/output (and nonce VRF for TPraos) through to the consensus `HeaderBody`. `verify_block_vrf` extracts VRF key + leader proof from each era's block header and delegates to `verify_leader_proof`. `VrfVerificationParams` bundles epoch nonce, sigma, and active slot coefficient.
- Nonce evolution wiring: `apply_nonce_evolution` extracts the per-era VRF nonce contribution (TPraos `nonce_vrf` or Praos `vrf_result`) and prev_hash from a `MultiEraBlock` and feeds them to `NonceEvolutionState::apply_block`. Byron blocks are skipped. Runtime nonce state MUST be initialized from `ShelleyGenesisHash` via upstream `genesisHashToPraosNonce`, and TICKN MUST use Shelley genesis `extraEntropy`; using `NeutralNonce` as the initial Praos nonce diverges VRF validation from official `mkTPraosParams` / `initialChainDepState`. Rollback to `Origin` without an exact ChainDepState sidecar MUST restore this configured origin nonce state, not `NeutralNonce`. VRF verification failures should surface slot, era, and epoch nonce through `SyncError::VrfVerification` so live preview/mainnet diagnostics can identify the exact bad header.
- Rollback sidecar hardening: `sync.rs` persists node-owned ChainDepState bundles `[version=1, point, nonce_state|null, ocert_counters|null]` under `storage_dir/chain_dep_state/<slot-hex>.cbor` only when a ledger checkpoint lands, after the corresponding nonce and OpCert updates. Rollback, restart, and LSQ recovery use this exact slot-indexed bundle as the canonical source; persistent non-origin rollback fails closed when exact ChainDepState history is unavailable. Restart recovery also loads the persisted `stake_snapshots.cbor` ring before boundary-aware replay; persistent non-origin recovery must not reconstruct the Praos stake snapshot ring from the ledger state because that can validate subsequent VRF proofs against the wrong epoch context. Runtime resume must carry the boundary-aware recovered `StakeSnapshots` and current-epoch pool block counts into `LedgerCheckpointTracking`, not reinitialize them from an empty map.
- Block body hash verification: `verify_block_body_hash` computes Blake2b-256 of body elements and compares against the hash declared in the header. `extract_header_block_body_hash` handles both 14-element (Praos) and 15-element (Shelley) header bodies. Wired into `sync_batch_apply_verified` via `VerificationConfig.verify_body_hash`.
- Block header hash computation uses real Blake2b-256.
- Mempool sync eviction: `extract_tx_ids` + `evict_confirmed_from_mempool`. TxSubmission runtime wiring now includes `serve_txsubmission_request_from_mempool`, a thin one-shot helper that answers a single `TxSubmissionClient` request from fee-ordered mempool state, plus `serve_txsubmission_request_from_reader` and `run_txsubmission_service`, which use an upstream-aligned `TxSubmissionMempoolReader`/`MempoolSnapshot` flow with a monotonic `last_idx` cursor so managed outbound serving only advertises transactions newer than the last acknowledged snapshot position. `run_txsubmission_service_shared` mirrors that managed flow for `SharedMempool` so the long-lived service can observe concurrent inserts between requests. For inbound-style admission, `add_tx_to_mempool`/`add_tx_to_shared_mempool` stage `LedgerState.apply_submitted_tx()` before committing the mempool insert, and `add_txs_to_mempool`/`add_txs_to_shared_mempool` repeat that logic in order so later transactions in a batch can depend on earlier accepted outputs while returning an explicit `MempoolAddTxResult` per transaction.
- Verified sync service: `run_verified_sync_service` with `VerifiedSyncServiceConfig` (batch_size + verification + optional nonce config + optional security_param + `LedgerCheckpointPolicy`) and `VerifiedSyncServiceOutcome` (includes final `NonceEvolutionState`, optional `ChainState`, and `stable_block_count`). Verified execution is split into fetch-only `sync_batch_verified` plus a ChainDb apply phase so coordinated storage mutation can happen after network I/O. `run_verified_sync_service_chaindb` now provides the same verified batch flow over coordinated storage so stable entries can be promoted through `yggdrasil-storage::ChainDb`; node still decides checkpoint cadence via `LedgerCheckpointPolicy`, but the underlying checkpoint clear, truncate, save, and retain mechanics now go through crate-owned `ChainDb` helpers. The traced reconnecting ChainDb runtime path emits `Node.Recovery.Checkpoint` events describing whether a checkpoint was persisted, skipped by cadence, or cleared, along with retention counts. Runtime also exposes `run_reconnecting_verified_sync_service_chaindb`, `run_reconnecting_verified_sync_service_chaindb_with_tracer`, `resume_reconnecting_verified_sync_service_chaindb`, and shared-ChainDb resume variants; CLI `run` now opens file-backed immutable, volatile, and ledger stores under `storage_dir`, can optionally bind `inbound_listen_addr`, and starts through the coordinated-storage resume path by default.
- Low-level coordinated-sync API note: `run_verified_sync_service_chaindb` now requires an explicit `base_ledger_state` argument instead of implicitly using `LedgerState::new(Era::Byron)`. Callers that want genesis-derived protocol parameters in replay validation must pass a genesis-seeded base state (for CLI/runtime this comes from `base_ledger_state(file_cfg, config_base_dir)` in `main.rs`).
- Recovery now includes `recover_ledger_state_chaindb`, a thin node-facing wrapper over storage-owned `ChainDb::recover_ledger_state()`, plus `recover_ledger_state_chaindb_epoch_boundary()` for epoch-aware replay from a genesis-seeded base state. Runtime resume paths use the boundary-aware helper when nonce/epoch tracking is configured, with the same `CekPlutusEvaluator` used by verified sync; governor ledger-peer refresh also receives the configured `EpochSchedule` so PPUP staging is interpreted at hard-fork boundaries. The storage crate owns typed checkpoint restore plus immutable/volatile suffix replay, while runtime still owns checkpoint cadence and reconnect orchestration. When a legacy checkpoint is missing genesis-derived Plutus cost models but the configured genesis provides them, recovery discards that checkpoint and replays from base storage rather than mutating a stale snapshot. Runtime recovery now returns the replayed `StakeSnapshots` plus `pool_block_counts`; callers must seed `LedgerCheckpointTracking` with those values so mid-epoch restarts preserve upstream `nesBcur` reward inputs. Both `run_reconnecting_verified_sync_service_chaindb` and `resume_reconnecting_verified_sync_service_chaindb` refresh the checkpoint store after successful verified batches.
- Runtime request helpers: `ReconnectingVerifiedSyncRequest::new(...)` and `ResumeReconnectingVerifiedSyncRequest::new(...)` now provide default-initialized constructors for optional reconnect fields while still requiring explicit base ledger state. Prefer these constructors in tests/helpers to keep reconnect call sites concise and to avoid silently dropping required base-state wiring.
- Runtime request helper usage: `main.rs` now builds its shared-ChainDb resumed sync request via `ResumeReconnectingVerifiedSyncRequest::new(...)` and then sets only non-default fields (`nonce_state`, `use_ledger_peers`, `peer_snapshot_path`, `metrics`), mirroring test usage and reducing manual request boilerplate.
- Fluent setter usage: request helpers now expose `with_nonce_state(...)`, `with_use_ledger_peers(...)`, `with_peer_snapshot_path(...)`, `with_metrics(...)`, `with_peer_registry(...)`, and `with_mempool(...)` so call sites can compose optional fields via method chaining rather than mutable field assignment.
- Reconnecting sync request helpers now also expose `with_tentative_state(...)` so runtime call sites can share diffusion pipelining tentative-header state with verified batch loops.
- Epoch boundary wiring: `advance_ledger_with_epoch_boundary()` in sync.rs detects epoch transitions during block application using `is_new_epoch()` / `slot_to_epoch()` and calls `apply_epoch_boundary()` before the first block of each new epoch. `LedgerCheckpointTracking` optionally carries `StakeSnapshots` + `EpochSize`; when present, `update_ledger_checkpoint_after_progress` uses epoch-aware advancement instead of plain `advance_ledger_state_with_progress`. Epoch boundary is automatically enabled when `nonce_config` provides `epoch_size`. `LedgerCheckpointTracking` also carries `pool_block_counts: BTreeMap<PoolKeyHash, u64>` to accumulate per-pool block production within each epoch; `advance_ledger_with_epoch_boundary` hashes each block's issuer VKey (Blake2b-224) to a `PoolKeyHash` and increments the count, then at epoch transitions calls `compute_pool_performance()` to derive actual `UnitInterval` performance ratios from the `set` stake snapshot before passing them to `apply_epoch_boundary()`. Counts are cleared after each epoch boundary; rollback recovery truncates checkpoints at the actual rollback point, restores the latest checkpoint at or before that point, and replays immutable+volatile blocks through the same epoch-boundary-aware path starting from the tracked/persisted `StakeSnapshots` ring when snapshot tracking is enabled. Both `advance_ledger_state_with_progress` and `advance_ledger_with_epoch_boundary` accept `evaluator: Option<&dyn PlutusEvaluator>` and call `apply_block_validated()`. Checkpoint-tracked replay now carries a stored `CekPlutusEvaluator` inside `LedgerCheckpointTracking`, seeded from `VerifiedSyncServiceConfig.plutus_cost_model` when available, so rollback replay and epoch-boundary advancement use the same genesis-derived CEK costs instead of recreating `CekPlutusEvaluator::new()` for each batch.
- **Plutus script evaluation** (`plutus_eval.rs`): `CekPlutusEvaluator` implements `yggdrasil_ledger::plutus_validation::PlutusEvaluator` using the `yggdrasil-plutus` CEK machine. It decodes raw on-chain `PlutusBinary` bytes via `decode_script_bytes` / `decode_script_bytes_allowing_remainder` as upstream `SerialisedScript` (CBOR bytestring containing Flat), applies `datum` (if spending), `redeemer`, and a version-aware `ScriptContext` derived from the normalized ledger `TxContext`, evaluates within the `ExUnits` budget declared by the transaction, and returns `Ok(())` for V1/V2 non-error outcomes or requires `Bool(true)` for V3. `is_script_well_formed()` mirrors upstream `validScript` / `decodePlutusRunnable`: enforce language activation when `ProtocolParameters.protocol_version` is present (`V1` PV5, `V2` PV7, `V3` PV9), then decode `PlutusBinary` deterministically (V1/V2 allow upstream's historical trailing remainder after the first script CBOR item; V3 rejects remainders). **ScriptContext encoding parity** (upstream `Cardano.Ledger.Alonzo.Plutus.TxInfo` / `Cardano.Ledger.Conway.TxInfo`): V1/V2 fee uses nested Value (`transCoinToValue`), V3 fee is plain Lovelace integer (`transCoinToLovelace`). V1/V2 mint prepends zero-ADA entry (`transMintValue = transCoinToValue zero <> transMultiAsset`), V3 mint does not. V1 datums/withdrawals are `List`-of-tuples (PV1 encoding), V2/V3 use `Map`. V1 TxOut is always 3-element `(address, value, maybe_datum_hash)` via `transTxOutV1`; V2/V3 Babbage TxOut is 4-element `(address, value, datum_option, script_ref)` via `transTxOutV2`. Validity intervals are protocol-aware: Alonzo/Babbage upper-only intervals encode an inclusive upper bound (`PV1.to`), bounded intervals use `strictUpperBound`, and Conway PV9+ upper-only intervals are strict. V1 guard rejects inline datums (`InlineDatumsNotSupported`) and reference scripts (`ReferenceScriptsNotSupported`) in inputs/outputs. `TxInfo` now includes resolved inputs/reference inputs, outputs with structured Shelley-family addresses, fee, mint, withdrawals (V2 staking-credential keyed, V3 credential keyed), certificates, signatories, redeemers, datums, tx id, Conway votes/proposals, and treasury fields. Byron/reward addresses are omitted from TxOut translation rather than encoded as bogus raw bytes. **PV9 Conway bootstrap phase parity** (upstream bug #4863): `tx_cert_data_v3()` accepts `protocol_version` from `TxContext`; when PV major == 9 (`is_conway_bootstrap_phase()`), `AccountRegistrationDeposit` and `AccountUnregistrationDeposit` omit the deposit/refund field (`Nothing` instead of `Just deposit`), matching upstream `hardforkConwayBootstrapPhase` in `Cardano.Ledger.Conway.TxInfo.transTxCert`. PV >= 10 includes the deposit normally. `script_purpose_data_v3()` and `script_info_data_v3()` also thread `protocol_version` for Certifying purpose encoding. The V3 certifying path translates the supported V3 `TxCert` forms we can derive from the typed ledger certificate set and now fails explicitly on unsupported certificates such as `GenesisDelegation` instead of fabricating integer placeholders; the proposing path likewise fails explicitly if a `ProposalProcedure` cannot be translated (for example malformed reward-account bytes). - **On-wire byte preservation + one-shot span cache through the block converter** (`sync.rs`, `runtime.rs`): all four era converters (`shelley_block_to_block`, `alonzo_block_to_block`, `babbage_block_to_block`, `conway_block_to_block`) plus `multi_era_block_to_block` populate each `Tx { body, witnesses }` from the on-wire byte spans returned by [`yggdrasil_ledger::extract_block_tx_byte_spans`](../../crates/ledger/src/cbor.rs) instead of from `to_cbor_bytes()` re-serialisation.  CBOR re-encoding is byte-canonical but does NOT always agree byte-for-byte with the on-wire encoding (definite vs indefinite length, set vs array, integer-width canonicalisation), and `min_fee = a · txSize + b` is sensitive to that drift.  Surfaced 2026-04-27 by a 30-min preprod sync rehearsal that crashed at slot ~518 460 (Byron→Shelley boundary) with `FeeTooSmall { minimum: 208269, declared: 207829 }` — a 440-lovelace gap (10 bytes × 44 a-coefficient).  Babbage-family min-UTxO has the same byte-preservation requirement at the transaction-body output level: the ledger extracts raw `TxOut` sizes from the original body because legacy array outputs re-encode as two-byte-larger post-Alonzo maps, while upstream `allSizedOutputsTxBodyF` uses the original `Sized TxOut`.  Each converter has both a public `*_block_to_block(block, raw_bytes)` form (extracts spans on the fly; used by external callers and one-off conversions) AND a public `*_block_to_block_with_spans(block, &spans)` form (reads pre-extracted spans; used on the sync hot path).  The three Alonzo-family `*_with_spans` helpers are generated by a single `alonzo_family_block_to_block_with_spans!` `macro_rules!` (60-line bodies are otherwise identical modulo era tag and typed block).  `MultiEraSyncStep::RollForward` carries `raw_blocks: Vec<Vec<u8>>` AND a parallel `block_spans: Vec<BlockTxRawSpans>` populated once at step construction by `extract_spans_per_block(&blocks, &raw_blocks)`; all three roll-forward consumers — mempool eviction (`extract_tx_ids` now takes `Option<&BlockTxRawSpans>` not raw bytes), volatile-store apply (`apply_multi_era_step_to_volatile`), and ledger-state advance via `advance_ledger_state_with_progress` (now using the spans-aware closure threaded through `for_each_roll_forward_block`) — read the cached spans instead of independently re-walking the block CBOR.  Before this caching: 3 extractions per block.  After: 1.  `fetch_range_blocks_typed` and `sync_step_multi_era` use `BlockFetchClient::request_range_collect_points_raw_with` (the existing raw+decoded API) so the same bytes peers gave us are the bytes we feed to the converter.  Test-only paths construct synthetic blocks and may pass `Vec::new()` for both `raw_blocks` and `block_spans`; the converter falls back to `to_cbor_bytes()` re-encoding which is fine for tests.  References: `Cardano.Ledger.Shelley.Tx.minfee`, `Cardano.Ledger.Core.txIdTxBody`, `Cardano.Ledger.Babbage.TxOut.babbageMinUTxOValue`.  Full writeup in [`docs/REAL_PREPROD_POOL_VERIFICATION.md`](../../docs/REAL_PREPROD_POOL_VERIFICATION.md).
