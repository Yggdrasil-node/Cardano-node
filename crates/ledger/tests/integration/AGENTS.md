# Guidance for the folder-backed ledger integration test crate.
Keep modules in this folder focused on a single era family or ledger-state subdomain.

## Scope
- Folder-backed integration modules loaded from `tests/integration.rs` via `tests/integration/mod.rs`.
- Era codec and block-envelope regression tests.
- Ledger-state behavior tests split by operational domain.

## Rules *Non-Negotiable*
- Prefer new files when a module starts mixing unrelated eras or ledger subsystems.
- Keep era boundaries explicit: Byron, Shelley, Allegra/Mary, Alonzo, Babbage, Conway, and Praos block-envelope tests should not collapse back into one catch-all file.
- Keep ledger-state tests grouped by behavior such as era application, stake and DRep state, committee state, or pool/reward/query behavior.
- Shared helpers MUST remain scarce, obvious, and `pub(super)` when cross-module access is required.
- Module names MUST describe the rule surface being protected, not the implementation phase that added them.
- Always update this file when the integration module layout changes.

## Current Phase
- The integration test crate is organized into focused modules for core CBOR, Shelley, Allegra/Mary, Alonzo, Byron, Babbage, Conway, Praos block envelopes, governance updates, Plutus/script codecs, multi-era UTxO behavior, ledger-state subdomains, witness validation, and BBODY size accounting.
- `block_body_size.rs`: 2 tests locking BBODY `validateMaxBlockBodySize` parity for full serialized transaction-byte accounting (witness/aux/is_valid included) in both single-tx and multi-tx aggregation paths.
- `witness_validation.rs`: end-to-end `apply_block()` validation coverage for VKey witness sufficiency, Ed25519 signature verification, native script evaluation, and Conway governance guardrails including bootstrap proposal allow-reject paths (`HardForkInitiation`, `ParameterChange`, and `InfoAction` acceptance plus non-bootstrap rejection), bootstrap vote allow-reject paths (DRep `InfoAction`, committee/SPO bootstrap-action acceptance, and non-bootstrap rejection), post-bootstrap voter permission tests (SPO security-group `ParameterChange` boundary: accepted when update touches security params, rejected when only non-security params), voter-existence checks, and governance-action validation. E2E governance lifecycle tests cover vote recast overwrites (Yes→No via separate blocks), proposal-and-vote in same block (intra-block visibility), cross-block lineage chain (HardFork v10→v11 via separate blocks), cert+proposal+vote combo (stake registration + proposal + vote in single block), and multiple proposals in one tx (distinct gov_action_index indexing). All signing-based tests use real Ed25519 signing via `yggdrasil_crypto::ed25519::SigningKey`.
- `plutus_evaluation.rs`: 6 tests covering `apply_block_validated()` with mock evaluators across Alonzo/Babbage/Conway, including minting-policy Plutus script dispatch, evaluator failure propagation, evaluator metadata assertions, and the no-evaluator soft-skip path via `apply_block()`.
- `missing_redeemer.rs`: 2 tests covering the missing-redeemer half of upstream `hasExactSetOfRedeemers`. Block and submitted-tx minting cases now reject required Plutus V1/V2 purposes that omit their redeemer pointer, exercising the shared `validate_plutus_scripts()` path before cost-model checks or CEK evaluation.
- `script_data_hash.rs`: 5 tests covering local `PPViewHashesDontMatch` parity. In addition to the Alonzo submitted-tx baseline, Babbage and Conway reference-input cases now assert that script-integrity hashing includes needed reference-script language views, excludes unused provided scripts from language views (upstream `mkScriptIntegrity` / `Map.restrictKeys scriptsProvided scriptsNeeded`), and preserves the era split between legacy redeemer encoding (Babbage) and Conway map-format redeemers.
- `submitted_script_witness.rs`: submitted-tx (`apply_submitted_tx`) script-witness coverage for Alonzo/Babbage paths, including rejection when a required script hash is missing from all witness sources and acceptance when the required hash is provided by a Babbage reference-input script reference.
- `invalid_metadata.rs`: 10 tests covering upstream `InvalidMetadata` / `validMetadatum` soft-fork gating. Oversized bytes/text metadatum values (>64 bytes) are rejected from Allegra onward in both block and submitted-tx paths, including nested array/map cases; exact-64-byte boundaries, empty bytes, large integers, and PV 2.0 soft-fork bypass cases are accepted.
- `no_cost_model.rs`: 9 tests covering upstream `NoCostModel` collect errors from `Cardano.Ledger.Alonzo.Plutus.Evaluate`. Block and submitted-tx paths reject missing Plutus V1/V2/V3 cost models by language key (0/1/2) before CEK evaluation when protocol params carry a `cost_models` map, and soft-skip the check when `cost_models` is absent.
- `network_validation.rs`: 12 tests covering upstream `WrongNetwork`, `WrongNetworkWithdrawal`, and `WrongNetworkInTxBody` rules. 8 block-path tests via `apply_block_validated()` across Shelley, Alonzo, and Babbage eras (output address mismatch, withdrawal account mismatch, tx body network_id mismatch, correct-network acceptance, skip-when-unset, absent-network-id acceptance). 4 submitted-tx-path tests via `apply_submitted_tx()` covering Shelley wrong-network output, Alonzo wrong-network-in-tx-body, Conway wrong-withdrawal-network, and Babbage missing-reference-input rejection.
- `conway_cert_deposit_validation.rs`: 15 tests covering upstream DELEG/GOVCERT deposit validation rules — key deposit mismatch phase split (`IncorrectDepositDELEG` during bootstrap PV 9, `DepositIncorrectDELEG` post-bootstrap PV 10+, including combined `AccountRegistrationDelegationTo*` variants), key refund mismatch phase split (`IncorrectKeyDepositRefund` bootstrap, `RefundIncorrectDELEG` post-bootstrap), `DrepIncorrectDeposit` (DRep deposit vs ppDRepDeposit), `DrepIncorrectRefund` (DRep refund vs stored deposit), `WithdrawalNotFullDrain` (Conway exact-drain withdrawal semantics), Shelley backward-compatibility (legacy registration/deregistration certs accepted without deposit validation in Conway mode).
- `dormant_epoch.rs`: 6 tests covering upstream `numDormantEpochs` tracking — CBOR round-trip (22-element LedgerState array), epoch-boundary increment when no governance proposals remain, epoch-boundary reset when proposals exist (with DRep-stake-seeded ratification that prevents premature ratification), per-tx DRep expiry bump via `updateDormantDRepExpiries` when proposals appear, dormant counter preservation for proposal-less transactions, skip logic for fully expired DReps whose bumped expiry would still not reach current epoch. Additionally validates Conway proposal deposit inclusion in value preservation equation (`totalTxDeposits = certDeposits + proposalDeposits`).
- `conway_governance_parity.rs`: 26 tests covering nine Conway parity fixes: (1) `UnelectedCommitteeVoters` — unelected hot credential rejected at PV ≥ 10, authorized voter accepted, PV < 10 bypass, resigned member's hot cred rejected; (2) `DelegateeDRepNotRegistered` — delegation to unregistered DRep returns distinct error variant, built-in DReps (`AlwaysAbstain`, `AlwaysNoConfidence`) succeed without registration; (3) Conway `AccountUnregistrationDeposit` (tag 8) enforces reward-balance check — rejects non-zero rewards (same as Shelley tag 1, per upstream `StakeKeyHasNonZeroAccountBalanceDELEG`), Shelley tag 1 still rejects, zero-balance baseline passes; (4) DRep unregistration clears delegations — `clearDRepDelegations` zeroes DRep delegation on all stakers delegated to the unregistered DRep, does not affect stakers delegated to other DReps; (5) Committee `isPotentialFutureMember` — authorization/resignation succeeds for cold credentials appearing in pending `UpdateCommittee` proposals, unknown-without-proposal fails with `CommitteeIsUnknown`; (6) `ZeroTreasuryWithdrawals` bootstrap gate — zero-sum treasury withdrawal proposal blocked during bootstrap (PV < 10) by `DisallowedProposalDuringBootstrap`, rejected after bootstrap (PV ≥ 10) by `ZeroTreasuryWithdrawals`, non-zero withdrawal accepted post-bootstrap; (7) Bootstrap DRep delegation (`preserveIncorrectDelegation`) — during bootstrap (PV 9), delegation to unregistered DRep succeeds for `DelegationToDrep`, `DelegationToStakePoolAndDrep`, and `AccountRegistrationDelegationToDrep`, rejected at PV 10; (8) Bootstrap DRep expiry (`computeDRepExpiryVersioned`) — dormant epoch subtraction skipped during bootstrap in `apply_conway_votes` and `touch_drep_activity_for_certs`; (9) Conway withdrawal DRep delegation (`validateWithdrawalsDelegated`) — post-bootstrap key-hash withdrawals require an existing DRep delegation, bootstrap bypasses the check, and script-hash withdrawals are excluded.
