Focus on core ledger plumbing shared across eras: CBOR codec, core types, and state integration surfaces.

## Scope
- `cbor.rs`, `types.rs`, generic ledger state helpers, and module wiring under `crates/ledger/src`.
- Boundaries between shared ledger infrastructure and `eras/` era-specific logic.

##  Rules *Non-Negotiable*
- Keep CBOR behavior deterministic and round-trip tested.
- Do not duplicate era-specific rules in shared modules.
- Maintain strong type wrappers for protocol-relevant identifiers (`SlotNo`, `BlockNo`, `HeaderHash`, `Point`, `TxId`).
- Public shared APIs MUST have Rustdocs when semantics are non-obvious.
- Stay true to the official type naming and terminology for node concepts, network protocols, and ledger types when possible.
- Always read the folder specific `**/AGENTS.md` files. They MUST stay current and MUST remain operational rather than long-form documentation. If the folder context is outdated, missing, or incorrect, update the relevant AGENTS.md file.

## Official Upstream References *Always research referances and add or update links as needed*
- Ledger repository: <https://github.com/IntersectMBO/cardano-ledger>
- Formal specs: <https://github.com/IntersectMBO/formal-ledger-specifications>

## Current Phase
- Hand-rolled CBOR encoder/decoder supports major Cardano-required primitives including signed integers (`integer()`).
- Shared typed core identifiers and point/nonce primitives are in place.
- Shared transaction plumbing now includes `compute_tx_id` plus submitted-transaction wrappers for Shelley-family and Alonzo-family wire shapes, with `MultiEraSubmittedTx` as the era-directed decode boundary for node-to-node relay work.
- Credential and address types landed: `StakeCredential` (key-hash/script-hash), `RewardAccount` (29-byte structured), `Address` (Base/Enterprise/Pointer/Reward/Byron variants), with CBOR codecs and variable-length natural encoding for pointer addresses. Deep validation now rejects invalid Shelley network ids, rejects pointer addresses with trailing bytes after the 3-variable-nat pointer, and exposes strict Byron bootstrap-address validation with outer `[tag24(payload), crc32]` structure and CRC32 verification via `Address::validate_bytes()` / `Address::validate()`.
- Certificate hierarchy landed in `types.rs`: `Anchor` (moved from conway.rs), `UnitInterval` (tag-30 rational), `Relay` (3-variant), `PoolMetadata`, `PoolParams` (9-field inline group), `DRep` (4-variant Conway), `DCert` (19-variant flat enum covering Shelley tags 0–5 and Conway tags 7–18), all with CBOR codecs in `cbor.rs`.
- `LedgerState` owns dual UTxO sets: `ShelleyUtxo` (legacy, authoritative for Shelley blocks) and `MultiEraUtxo` (generalized). It also carries explicit `PoolState`, `StakeCredentials`, `CommitteeState`, `DrepState`, `RewardAccounts`, and `current_epoch: EpochNo` containers plus `LedgerStateSnapshot` read-only views and `LedgerStateCheckpoint` restorable checkpoints. `apply_block()` dispatches per era with atomic block application; Byron blocks now perform real UTxO transitions via `apply_byron_block()` which decodes `ByronTx` from transaction body bytes, validates input existence and non-negative implicit fee (`consumed >= produced`), and applies UTxO updates atomically on `multi_era_utxo`. Byron inputs (`ByronTxIn` with u32 index) are converted to `ShelleyTxIn` (u16 index) for unified UTxO set storage; Byron outputs are stored as `MultiEraTxOut::Shelley` since their shape is identical to `ShelleyTxOut`. Shelley block/submitted-tx application now rebuilds `multi_era_utxo` from the authoritative `ShelleyUtxo` after each successful transition so Shelley outputs survive the Shelley→Allegra handoff. Transient pending Shelley genesis bundles can also be attached to `LedgerState`; initial funds and static stake delegations both activate once on the first Shelley-family block and are intentionally omitted from checkpoint CBOR so persisted snapshots remain pure ledger state. `RegisteredPool::relay_access_points()` and `PoolState::relay_access_points()` now expose only directly dialable single-host relay forms with declared ports so higher layers can consume immutable-ledger peers without re-implementing relay decoding.
- `LedgerStateCheckpoint`, `LedgerState`, the state-container wrappers, and both UTxO views now have deterministic CBOR round-trip support so higher layers can persist typed recovery checkpoints instead of opaque ad hoc bytes.
- `utxo.rs` module landed: `MultiEraTxOut` enum (Shelley/Mary/Alonzo/Babbage), `MultiEraUtxo` with per-era apply methods including Byron (`apply_byron_tx()` with implicit fee validation and ByronTxIn→ShelleyTxIn conversion), TTL, validity interval start, coin preservation, and multi-asset preservation checks.
- Era-specific structures live under `eras/`; all eras Shelley through Conway are implemented. Shared layer should stay lightweight and stable.
- `PlutusData` is integrated into: `Redeemer.data` (typed payload), `DatumOption::Inline` (typed inline datum with tag-24 double encoding), `ShelleyWitnessSet.plutus_data` (typed `Vec<PlutusData>`).
- `protocol_params.rs`: `ProtocolParameters` struct with all Shelley-through-Conway parameter fields, `Default` (Shelley mainnet), `alonzo_defaults()`, `min_lovelace_for_utxo()`, full CBOR map-based round-trip codec. `ProtocolParameterUpdate` is now a typed sparse delta with matching CBOR-map keys, `is_empty()`, and `ProtocolParameters::apply_update()`. Wired into `LedgerState` as `Option<ProtocolParameters>` (array element 10, backward-compatible with 9-element legacy).
- `fees.rs`: Fee calculation and validation — `min_fee_linear()`, `script_fee()`, `total_min_fee()`, `validate_fee()`, `validate_tx_ex_units()`, `validate_tx_size()`.
- `native_script.rs`: Native (timelock) script evaluator — `evaluate_native_script()`, `native_script_hash()` (Blake2b-224 with language tag prefix), `NativeScriptContext`.
- `collateral.rs`: Alonzo+ collateral validation — `validate_collateral()` checks count limit, UTxO lookup, non-ADA rejection, and percentage-of-fee sufficiency.
- `min_utxo.rs`: Minimum UTxO output validation — `validate_min_utxo()`, `validate_all_outputs_min_utxo()` using protocol params.
- `witnesses.rs`: VKey witness sufficiency and Ed25519 signature verification — `validate_vkey_witnesses()`, `verify_vkey_signatures()` (real Ed25519 verification against tx body hash using `yggdrasil_crypto::ed25519`), `verify_bootstrap_witnesses()` (Ed25519 verification for bootstrap witnesses plus CBOR-map validation of Byron address attributes), `vkey_hash()` (Blake2b-224), `witness_vkey_hash_set()`. Helper functions for collecting required VKey hashes and script hashes from spending inputs, certificates, withdrawals, and Conway voting procedures — both Shelley UTxO and multi-era UTxO variants. All per-era `apply_block()` inner loops now call `validate_witnesses_if_present()` to enforce both VKey witness hash sufficiency and witness signature validity when witness bytes are provided.
- `witnesses.rs` now also exposes `required_script_hashes_from_mint()` so mint policy IDs participate in required-script collection for Mary, Alonzo, Babbage, and Conway block application.
- `plutus_validation.rs`: trait-based Plutus phase-2 seam for the ledger crate — `PlutusEvaluator`, `PlutusScriptEval`, `PlutusVersion`, `ScriptPurpose`, `plutus_script_hash()`, and `validate_plutus_scripts()`. Spending-purpose Plutus resolution now uses the staged `MultiEraUtxo` to recover the spent output's script hash from the payment credential and the datum from either witness-set datum hashes (Alonzo-style) or inline datums (Babbage/Conway-style). Full `ScriptContext` / `TxInfo` construction is still a future milestone. Use this path instead of adding a direct dependency from `ledger` to `plutus`.
- Per-era `apply_block()` native script validation: Allegra, Mary, Alonzo, Babbage, and Conway inner loops call `validate_native_scripts_if_present()` to evaluate required native scripts. Required-script collection now covers script-hash payment credentials in spending inputs, certificates, withdrawals, and mint policy IDs for mint-capable eras. Missing native scripts are silently skipped because they may instead be Plutus scripts resolved by `validate_plutus_scripts()`.
- `state.rs` now exposes `LedgerState::apply_block_validated()` as the evaluator-aware block-application entry point. Higher layers that have a Plutus CEK implementation should call this method and inject a `PlutusEvaluator`; `apply_block()` remains the no-evaluator convenience wrapper.
- Conway block application now validates `current_treasury_value` against `LedgerState.accounting.treasury` when supplied, validates proposal deposits against `ProtocolParameters.gov_action_deposit` when present, rejects `TreasuryWithdrawals` proposals whose requested withdrawals sum to zero, rejects `UpdateCommittee` proposals that add and remove the same cold credential via `ConflictingCommitteeUpdate`, rejects `UpdateCommittee` member additions whose expiration epoch is not strictly after `LedgerState.current_epoch` via `ExpirationEpochTooSmall`, rejects invalid previous governance-action references via `InvalidPrevGovActionId` both for same-transaction self and forward references and for cross-transaction references whose stored parent action is missing or has the wrong governance purpose, rejects votes on stored governance actions whose tracked `expires_after` is earlier than `LedgerState.current_epoch` via `VotingOnExpiredGovAction`, and rejects empty typed `ParameterChange.protocol_param_update` payloads via `MalformedProposal`. Enactment now applies typed parameter updates directly to `LedgerState.protocol_params` through `ProtocolParameters::apply_update()` and records the lineage root.
- Conway hard-fork proposals now enforce upstream `pvCanFollow` sequencing via `ProposalCantFollow`: the first `HardForkInitiation` is compared against `ProtocolParameters.protocol_version`, and chained hard-fork proposals are compared against the referenced stored hard-fork action's target protocol version.
- `ProtocolParameters` now carries `protocol_version`, seeded from Shelley genesis `protocolParams.protocolVersion` and encoded at the upstream protocol-parameter key 14. Protocol-parameter CBOR decode remains backward-compatible with older local checkpoints that wrote `min_utxo_value` at key 14 before protocol-version tracking was added.
- Conway bootstrap GOV checks are now enforced when `ProtocolParameters.protocol_version` has major version 9: proposals are limited to bootstrap actions (`ParameterChange`, `HardForkInitiation`, `InfoAction`), DReps may only vote on `InfoAction`, and committee/SPO voters may only vote on bootstrap actions. This remains a bounded slice until on-chain protocol-version evolution is modeled beyond the genesis-seeded baseline.
- Conway malformed-proposal validation now includes bounded `ParameterChange.protocol_param_update` guardrails beyond non-empty checks: UnitInterval fields must satisfy rational bounds, threshold bundles (`PoolVotingThresholds`/`DRepVotingThresholds`) are range-checked, selected positive-only fields reject zero, protocol-version major rejects zero, and tx/block-size constraints reject `max_tx_size > max_block_body_size` against either the proposed update pair or current protocol parameters. The local purpose grouping matches upstream Conway GOV roots: `ParameterChange`, `HardForkInitiation`, and `NewConstitution` each require the same constructor family, while `NoConfidence` and `UpdateCommittee` share the committee-purpose lineage. Stored `GovernanceActionState` now optionally carries `proposed_in` and `expires_after` metadata derived from `ProtocolParameters.gov_action_lifetime`; legacy checkpoints without that metadata still decode with `None`. Proposal and TreasuryWithdrawals reward-account network ids are validated against `LedgerState.expected_network_id` when configured, Conway voters are validated against the post-CERTS committee/DRep/pool state so same-tx committee hot-key authorization and DRep registration count for `VotersDoNotExist`, accepted proposals persist in `LedgerState.governance_actions`, accepted votes are recorded into the stored action state, and DRep unregistration removes stored DRep votes. Typed parameter-group classification for SPO threshold gating and full live protocol-version/enact-state tracking still need richer Conway governance state.
- `stake.rs`: Stake distribution snapshots and epoch-boundary snapshot rotation — `IndividualStake`, `Delegations`, `StakeSnapshot`, `StakeSnapshots` (three-snapshot ring: mark/set/go + fee pot), `PoolStakeDistribution`, `compute_stake_snapshot()`, `compute_drep_stake_distribution()`, `StakeSnapshots::rotate()`, `StakeSnapshot::pool_stake_distribution()`. All types have CBOR round-trip codecs. `compute_drep_stake_distribution()` aggregates per-DRep delegated lovelace from `StakeSnapshot` stake amounts and credential-level DRep delegations for use in ratification vote tallying.
- `rewards.rs`: Epoch reward calculation (Shelley spec Section 10) — `RewardParams`, `EpochRewardPot`, `PoolRewardBreakdown`, `EpochRewardDistribution`, `mul_rational()`, `compute_epoch_reward_pot()`, `max_pool_reward()`, `compute_pool_reward()`, `compute_epoch_rewards()`. Uses u128 fixed-point arithmetic with SCALE=10^12.
- `epoch_boundary.rs`: Epoch boundary orchestration (NEWEPOCH / EPOCH / SNAP / RUPD / RATIFY) — `apply_epoch_boundary()` performs stake snapshot rotation, reward distribution, pool retirement + deposit refunds, governance action expiry + deposit refunds, ratification + enactment of accepted governance actions, DRep inactivity detection, and treasury/reserves accounting update. `retire_pools_with_refunds()` captures reward accounts before removing pools. `remove_expired_governance_actions()` prunes proposals whose `expires_after` epoch has passed and refunds deposits to registered return accounts (matching upstream Conway EPOCH rule). `ratify_and_enact()` iterates surviving governance actions, evaluates each against committee/DRep/SPO acceptance predicates via `ratify_action()`, enacts accepted actions via `LedgerState::enact_action()`, and removes enacted actions from the pending set. When protocol parameters are present but the Conway voting-threshold fields are unset, ratification falls back to the built-in `PoolVotingThresholds::default()` / `DRepVotingThresholds::default()` values instead of silently skipping governance actions. DRep inactivity step computes the set of DReps exceeding the `drep_activity` window (upstream `drepExpiry`); inactive DReps remain registered but are excluded from ratification quorum calculations. Returns `EpochBoundaryEvent` summary with `governance_actions_expired`, `governance_deposit_refunds`, `expired_gov_action_ids`, `dreps_expired`, `governance_actions_enacted`, `enacted_gov_action_ids`, and `enact_outcomes`.
- Shelley PPUP (protocol-parameter updates) is wired in `state.rs` + `epoch_boundary.rs`: Shelley/Allegra/Mary/Alonzo/Babbage block application now collects `ShelleyUpdate` proposals into `LedgerState.pending_pparam_updates`; epoch transitions call `apply_pending_pparam_updates(new_epoch)` before reward math, requiring quorum (majority of active `gen_delegs`) on an identical `ProtocolParameterUpdate`. `EpochBoundaryEvent` includes `pparam_updates_applied` for observability.
- `state.rs` DRep activity tracking: `RegisteredDrep` carries `last_active_epoch: Option<EpochNo>` (3-element CBOR array, backward-compatible with legacy 2-element). `new_active()`, `last_active_epoch()`, `touch_activity()` methods. Activity is touched on DRep registration, DRep update (via `touch_drep_activity_for_certs()`), and vote casting (via `apply_conway_votes()`). `DrepState::inactive_dreps()` returns DReps whose `last_active_epoch + drep_activity < epoch`.
- `protocol_params.rs` now includes `drep_deposit: Option<u64>` (CDDL key 31) and `drep_activity: Option<u64>` (CDDL key 32) for Conway DRep governance parameters.
- `state.rs` deposit tracking: `DepositPot` (key_deposits, pool_deposits, drep_deposits) and `AccountingState` (treasury, reserves) with CBOR codecs. `LedgerState` now 12-field struct with deposit_pot and accounting. `apply_certificates_and_withdrawals()` takes 10 args (deposit_pot + key_deposit + pool_deposit) and tracks deposits across all certificate variants. `process_retirements(epoch)` on `PoolState` removes retiring pools.
- CBOR `Decoder::peek_is_null()` added for nullable field support.
- `state.rs` governance enactment: `EnactState` struct tracks the current enacted constitution, committee quorum, and four purpose-lineage prev-action-ids (PParamUpdate, HardFork, Committee, Constitution) matching upstream `GovRelation`. `enact_gov_action()` free function implements the Conway ENACT rule for all seven `GovAction` variants: InfoAction (no effect), NewConstitution (replace constitution + lineage), NoConfidence (remove all committee + reset quorum + lineage), UpdateCommittee (add/remove members + set quorum + lineage), HardForkInitiation (set protocol_version + lineage), TreasuryWithdrawals (credit registered reward accounts from treasury), ParameterChange (record lineage, opaque update deferred). Returns `EnactOutcome` enum for informational reporting. `LedgerState` now carries `enact_state: EnactState` (16-element CBOR, backward-compatible with 15-element). `LedgerStateSnapshot` mirrors the field. `enacted_root()` on `EnactState` returns the enacted root GovActionId for a given `ConwayGovActionPurpose`. `validate_conway_proposals()` uses enacted-root semantics: `prev_action_id = None` is valid only when the enacted root for that purpose is also `None`; `prev_action_id = Some(id)` must match either the enacted root or a stored pending proposal of the same purpose. `NoConfidence` and `UpdateCommittee` share the `Committee` purpose group.
- `state.rs` ratification tally engine (Phase 6): `VoteTally` struct with `meets_threshold()` using cross-multiplication to avoid floats. `tally_committee_votes()` counts CC yes/no/abstain by equal-weight registered non-resigned members. `tally_drep_votes()` counts DRep votes weighted by delegated stake, excluding inactive DReps past `drep_activity` window. `tally_spo_votes()` counts SPO votes weighted by pool stake from `PoolStakeDistribution`. `drep_threshold_for_action()` / `spo_threshold_for_action()` map `ConwayGovActionPurpose` to per-type thresholds from `DRepVotingThresholds` / `PoolVotingThresholds`. `accepted_by_committee()` / `accepted_by_dreps()` / `accepted_by_spo()` predicates; `ratify_action()` combines all three. InfoAction always accepted. `LedgerState::enact_action()` method wraps `enact_gov_action()` to avoid split-borrow issues when enacting within epoch boundary processing. `GovernanceActionState::record_vote()` records a voter's vote for testing and block application. Reference: `Cardano.Ledger.Conway.Rules.Ratify`.
- `protocol_params.rs` voting thresholds: `PoolVotingThresholds` (5 `UnitInterval` fields: motion_no_confidence, committee_normal, committee_no_confidence, hard_fork_initiation, pp_security_group) and `DRepVotingThresholds` (10 `UnitInterval` fields: motion_no_confidence, committee_normal, committee_no_confidence, update_to_constitution, hard_fork_initiation, pp_network_group, pp_economic_group, pp_technical_group, pp_gov_group, treasury_withdrawal) with CBOR array encode/decode. `ProtocolParameters` new fields: `pool_voting_thresholds` (key 25), `drep_voting_thresholds` (key 26), `min_committee_size` (key 27), `committee_term_limit` (key 28).
