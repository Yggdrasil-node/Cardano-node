# Guidance for Cardano mini-protocol and peer networking work
Focus on typed protocol state machines, connection lifecycle, and exact wire-behavior boundaries.

## Scope
- Handshake, multiplexing, mini-protocol state machines, and peer lifecycle.
- Peer candidate resolution, topology domain types, and bootstrap-target ordering helpers that feed runtime peer policy.
- Node-to-node and node-to-client protocol surfaces.

##  Rules *Non-Negotiable*
- Handshake and multiplexing interfaces MUST remain independent from peer policy logic.
- Topology domain types, peer candidate ordering helpers, and the governor decision engine (promotion/demotion/churn policy) belong here. The governor is a pure decision function — effectful connection management stays in `node`.
- Mini-protocols MUST be introduced incrementally, starting with ChainSync and BlockFetch.
- Testable state transitions MUST be preferred over implicit runtime behavior.
- Public protocol types, handshake surfaces, and state-machine functions MUST have Rustdocs where message flow or invariants are not self-evident.
- Stay true to the official type naming and terminology for node concepts, network protocols, and ledger types when possible.
- Naming MUST mirror the official node and Ouroboros network specs so protocol traces and docs line up cleanly.
- Wire behavior and protocol sequencing MUST be explained by reference to the official node and upstream Ouroboros network sources.
- Always read the folder specific `**/AGENTS.md` files. They MUST stay current and MUST remain operational rather than long-form documentation. If the folder context is outdated, missing, or incorrect, update the relevant `AGENTS.md` file.

## Current Phase
- Multiplexer framing (SDU header encode/decode, MiniProtocolNum, MiniProtocolDir) is implemented and tested.
- Async bearer transport (Bearer trait, TcpBearer, Sdu framing) is implemented and tested over TCP loopback.
- Multiplexer/demuxer (`mux.rs`) is implemented: `start()` splits a TcpStream into reader/writer halves, spawns background demux (reads SDUs, dispatches by protocol number to per-protocol ingress channels) and mux (collects tagged payloads from protocol handles, frames as SDUs) tasks. Per-protocol `ProtocolHandle` provides `send()`/`recv()` for exchanging payloads through the mux. Tested: single-protocol round-trip, multi-protocol routing, bidirectional exchange, connection close detection, empty payloads, multiple messages, clean shutdown on handle drop.
- SDU segmentation is implemented for large protocol payloads: mux writer splits payloads into `MAX_SEGMENT_SIZE` chunks, and `MessageChannel` reassembles segmented SDUs into complete CBOR messages using self-delimiting CBOR item-length detection.
- Peer connection lifecycle (`peer.rs`) is implemented: `connect()` (initiator) and `accept()` (responder) establish TCP, start the mux, run the Handshake mini-protocol (ProposeVersions / AcceptVersion / Refuse), and return a `PeerConnection` with negotiated version + data-protocol handles. Version negotiation selects the highest common version matching network magic. Tested: happy-path connect/accept, data exchange after handshake, handshake refusal on magic mismatch, highest-version selection.
- Handshake protocol types, state machine, and CBOR wire codec (ProposeVersions/AcceptVersion/Refuse/QueryReply) are complete. `decode_version_data()` is version-aware: V7–V10 accept 2-element CBOR arrays `[networkMagic, initiatorOnlyDiffusionMode]`, V11–V12 accept 3-element `[..., peerSharing]`, V13+ accept 4-element `[..., peerSharing, query]`. Defaults: `peer_sharing=0`, `query=false` for shorter arrays.
- ChainSync, BlockFetch, KeepAlive, and TxSubmission2 have full message enums, validated state machines, transition tests, and CBOR wire codecs.
- Wire tags match upstream CDDL. ChainSync and BlockFetch now expose typed point/range helpers (`request_next_typed`, `find_intersect_points`, `request_range_points`) backed by ledger `Point` CBOR. ChainSync also exposes generic decoded-header support via `request_next_decoded_header::<H>()`. BlockFetch now exposes both per-item decode helpers (`recv_block_decoded::<B>()`, `recv_block_with()`, `recv_block_raw_with()`) and batch collection helpers (`request_range_collect*`) for raw, decoded, and raw+decoded flows. TxSubmission now uses ledger `TxId` for advertised/requested identifiers, exposes typed reply helpers for both storage `Tx` wrappers and ledger `MultiEraSubmittedTx` values, and tracks the outstanding/requestable TxId FIFO so invalid `ack`, blocking-mode, and request sequences are rejected locally while transaction bodies on the wire remain raw bytes.
- Server-side (responder) drivers are now implemented for all four data mini-protocols: `KeepAliveServer` (echo loop), `BlockFetchServer` (range-request + batch streaming), `ChainSyncServer` (request-next + find-intersect responding), and `TxSubmissionServer` (server-driven tx-id/tx request-reply). Each server driver wraps `MessageChannel`, maintains the protocol state machine, and provides typed send/receive methods mirroring the client driver pattern.
- Upstream planning anchor: `cardano-node` `TopologyP2P` and `ouroboros-network` `Diffusion.Topology` model local root groups with `hotValency`, `warmValency`, `diffusionMode`, and trustability, while `PublicRootPeers` distinguishes configured public roots, bootstrap peers, ledger peers, and big ledger peers. Yggdrasil now carries those local and public root topology domain types in `peer_selection.rs`.
- Yggdrasil also now carries network-owned topology root configuration and an initial resolved root-provider snapshot, including upstream-style `UseBootstrapPeers` and `UseLedgerPeers` semantics plus precedence and disjointness handling between local, bootstrap, and public roots.
- A minimal peer registry now also lives in this crate. It tracks per-peer `PeerSource` tags and `PeerStatus` state, and it can reconcile the canonical root-provider snapshot into root-peer registry entries without involving `node`.
- Root-set provider layer is now expanded: DNS-backed root-peer provider re-resolves configured local-root, bootstrap, and public-root access points with optional `DnsRefreshPolicy` (TTL clamping 60s/900s, exponential backoff). Provider refreshes reconcile the `PeerRegistry` on crate-owned paths.
- Peer registry is now extended with ledger, big-ledger, and peer-share source reconciliation helpers that preserve unrelated sources and peer status.
- Ledger peer provider layer is now complete: `LedgerPeerProvider` trait, `LedgerPeerSnapshot` normalization (deduplicates and enforces disjoint ledger/big-ledger sets), `LedgerPeerProviderRefresh` (combined/per-kind), `apply_ledger_peer_refresh()` helper, `refresh_ledger_peer_registry()` orchestration, `judge_ledger_peer_usage()` plus `reconcile_ledger_peer_registry_with_policy()` for `useLedgerAfterSlot` / latest-slot / ledger-state / peer-snapshot freshness gating, and `ScriptedLedgerPeerProvider` for testing. Provider refreshes reconcile the `PeerRegistry` on crate-owned paths without node involvement.
- Peer governor module (`governor.rs`) implements a pure decision engine: `GovernorTargets` (target_known/established/active plus big-ledger variants, upstream `sanePeerSelectionTargets` validation), `LocalRootTargets` (per-group warm/hot valency, `trustable` flag), `GovernorAction` (PromoteToWarm/PromoteToHot/DemoteToWarm/DemoteToCold/ForgetPeer/ShareRequest/RequestPublicRoots/RequestBigLedgerPeers/AdoptInboundPeer), evaluation functions for promotions/demotions/big-ledger-promotions/big-ledger-demotions/forget-cold-peers/forget-failed-peers/peer-share-requests, local-root valency enforcement, and combined `governor_tick`. Promotion functions deprioritize tepid peers (recently hot→warm demoted) matching upstream `knownPeerTepid` from `Ouroboros.Network.PeerSelection.State.KnownPeers`. Regular vs big-ledger target accounting is fully disjoint (`RegularPeerCounts` excludes `PeerSourceBigLedger`). `GovernorState` carries mutable failure-tracking with time-based exponential backoff and decay (`PeerFailureRecord`, `failure_decay`, `prune_decayed_failures`), `max_connection_retries` (upstream `reportFailures` `maxFail` — cold peers exceeding this threshold with only ephemeral sources are forgotten), in-flight promotion tracking (`in_flight_warm`/`in_flight_hot` — upstream `inProgressPromoteCold`/`inProgressPromoteWarm`), in-flight demotion tracking (`in_flight_demote_warm`/`in_flight_demote_hot` — upstream `inProgressDemoteWarm`/`inProgressDemoteHot`), peer-sharing request budget (`in_progress_peer_share_reqs`/`max_in_progress_peer_share_reqs` — upstream `inProgressPeerShareReqs`/`policyMaxInProgressPeerShareReqs`), and two-phase churn cycle (`ChurnPhase`: Idle → DecreasedActive → DecreasedEstablished → Idle; `apply_churn_to_targets()` lowers targets during decrease phases so standard evaluation functions produce bulk demotions; restored targets drive fresh promotions). `churn_decrease()` implements upstream `decrease v = max 0 $ v - max 1 (v div 5)`. Bootstrap-sensitive mode (`PeerSelectionMode::Sensitive`/`Normal`): `requires_bootstrap_peers()`, `peer_selection_mode()`, `is_node_able_to_make_progress()` implement upstream `Cardano.Network.PeerSelection.Bootstrap` logic; in sensitive mode `governor_tick` demotes non-trustable peers, filters promotions to trustable-only, and suppresses big-ledger promotions. `NodePeerSharing` enum (`PeerSharingDisabled`/`PeerSharingEnabled`) mirrors upstream `PeerSharing` willingness flag from handshake version data. `AssociationMode` enum (`LocalRootsOnly`/`Unrestricted`) mirrors upstream `AssociationMode` from `Ouroboros.Network.PeerSelection.Governor.Types`; computed via `compute_association_mode()` which implements upstream `readAssociationMode` logic from `cardano-diffusion`. In `LocalRootsOnly` mode (BP/hidden-relay topology), `governor_tick` in Normal mode suppresses both big-ledger peer promotions and peer sharing requests. Peer sharing requests are only generated in Normal+Unrestricted mode. `PeerRegistryEntry` now carries upstream `knownPeerTepid` (`tepid: bool`) — set on hot→warm demotion, cleared on cold→warm promotion. `evaluate_forget_failed_peers` forgets cold ephemeral peers whose decayed failure count exceeds `max_connection_retries`; protected sources (local-root, bootstrap, ledger, big-ledger) are never forgotten. `evaluate_forget_cold_peers` now enforces the one-sided `target_root` floor by only forgetting public-root peers when regular root count is above `target_root`. `evaluate_known_peer_discovery` mirrors upstream `KnownPeers.belowTarget`: coin-flip inbound-vs-peer-share selection with inbound retry gating (60s) and max 10 inbound peers per adoption round. `evaluate_peer_share_requests` generates `ShareRequest` actions when known < target_known and peer-sharing budget allows, excluding local-root, bootstrap, big-ledger, and cold peers. `filter_backed_off` now filters both duplicate promotions and duplicate demotions for peers with in-flight actions. `PeerSelectionCounters` (upstream `PeerSelectionView Int`) provides structured governor counters across four peer categories (regular, big-ledger, local-root, non-root) with in-flight action counts from `GovernorState`; `from_registry()` implements upstream `peerSelectionStateToView`. `OutboundConnectionsState` (`TrustedStateWithExternalPeers`/`UntrustedState`) mirrors upstream trust health signal; `compute_outbound_connections_state()` branches on `(AssociationMode, UseBootstrapPeers)` to determine whether all established connections are trustable. `FetchMode` (`FetchModeBulkSync`/`FetchModeDeadline`) mirrors upstream block-fetch concurrency mode; `fetch_mode_from_judgement()` derives mode from `LedgerStateJudgement`. `ChurnMode` (`Normal`/`BulkSync`) derived from `FetchMode` via `churn_mode_from_fetch_mode()`. `ChurnRegime` (`ChurnDefault`/`ChurnPraosSync`/`ChurnBootstrapPraosSync`) derived from `(ChurnMode, UseBootstrapPeers, ConsensusMode)` via `pick_churn_regime()`. `ChurnConfig` carries `bulk_churn_interval` (900 s) and `deadline_churn_interval` (3300 s) matching upstream defaults; `interval_for_mode(FetchMode)` selects the appropriate interval. Regime-aware churn: `churn_decrease_active()` and `churn_decrease_established()` vary aggressiveness by `ChurnRegime` (BootstrapPraosSync is most aggressive). `ConsensusMode` (`PraosMode`/`GenesisMode`) mirrors upstream. `PeerSelectionTimeouts` groups upstream policy time constants from `simplePeerSelectionPolicy`. `ConnectionManagerCounters` (upstream `ConnectionManagerCounters`) tracks 6 connection-state fields with `from_registry()` and field-wise `Add`. Randomized peer selection via `PickPolicy` (upstream `simplePeerSelectionPolicy` PickPolicy callbacks): `Xorshift64` embedded PRNG (no `rand` crate dependency), `PickPolicy::pick(count, candidates)` for uniform random subset selection, `PickPolicy::pick_scored(count, candidates, metrics)` for score-weighted selection (hot demotion). `PeerMetrics` tracks per-peer `upstreamyness`/`fetchyness` with `combined_score()` for hot demotion scoring (upstream `hotDemotionPolicy`). All evaluation functions now take `&mut PickPolicy` for randomized candidate selection; `evaluate_hot_to_warm_demotions` additionally takes `&PeerMetrics`. 171 governor tests.
- PeerSharing protocol (mini-protocol 10) is now implemented: `PeerSharingState` state machine, `PeerSharingMessage` (MsgShareRequest/MsgSharePeers/MsgDone), `SharedPeerAddress` (IPv4/IPv6 CBOR codec). Client driver `PeerSharingClient` and server driver `PeerSharingServer` (with serve_loop callback) are in place. 8 protocol-level tests.
- Node-to-Client handshake accept (`ntc_peer.rs`): `ntc_accept()` implements full NtC handshake negotiation — decodes `ProposeVersions`, negotiates best common version from `NTC_SUPPORTED_VERSIONS` (V16–V9), validates network magic, sends `AcceptVersion` or `RefuseVersionMismatch`, returns `NtcPeerConnection` with mux handle + negotiated `NodeToClientVersionData`. 5 NtC handshake tests.
- TCP listener (`listener.rs`) wraps `TcpListener` for inbound peer connections: `PeerListener::bind()`, `from_listener()`, `accept_peer()`.
- Connection manager type system (`connection.rs`) implements the upstream `ouroboros-network-framework` connection manager and inbound governor type foundations: `Provenance` (Inbound/Outbound), `DataFlow` (Unidirectional/Duplex), `TimeoutExpired`, `AbstractState` (12 upstream variants from `UnknownConnectionSt` through `TerminatedSt`), `ConnectionState` (10 runtime state-machine variants), `ConnectionType` (5 derived variants), `ConnStateId`, `ConnectionId` (local+remote SocketAddr), `connection_state_to_counters()` (upstream `connectionStateToCounters`), `verify_abstract_transition()` (full upstream state-machine transition validation — outbound path, inbound path, duplex, self-connect/overwrite races, terminal), `AcceptedConnectionsLimit` (512/384/5s defaults with linear `accept_delay()`), `MaybeUnknown<S>`, `Transition`, `OperationResult`, `DemotedToColdRemoteTr`, `ConnectionManagerError` (9 variants). Inbound governor types: `RemoteSt` (4 variants, `is_established()`), `InboundGovernorCounters` (cold/idle/warm/hot with `count_state()` and `Add`), `ResponderCounters`, `InboundGovernorEvent` (10 variants). Timeout constants module: `TIME_WAIT_TIMEOUT` 60s, `PROTOCOL_IDLE_TIMEOUT` 5s, `RESET_TIMEOUT` 5s, `SDU_TIMEOUT` 30s, `SDU_HANDSHAKE_TIMEOUT` 10s, `LOCAL_PROTOCOL_IDLE_TIMEOUT` 2s, `LOCAL_TIME_WAIT_TIMEOUT` 0s. Inbound constants: `MATURE_PEER_DELAY` 900s, `INACTION_TIMEOUT` ~31.4s. 49 connection tests.
- Inbound governor decision engine (`inbound_governor.rs`): pure step-function processing 10 `InboundGovernorEvent` variants into `InboundGovernorAction` (PromotedToWarmRemote/DemotedToColdRemote/ReleaseInboundConnection/UnregisterConnection). `InboundGovernorState` tracks per-connection `InboundConnectionEntry` (remote_st, data_flow, responder_counters, idle tracking), fresh/mature duplex peer maps, and counters. Handlers: new_connection (RemoteIdleSt + duplex maturation tracking), mux_finished (unregister), wait_idle_remote (warm/hot→idle), awake_remote (idle/cold→warm), promote/demote (warm↔hot), commit_remote (idle→release), matured_duplex_peers, inactivity_timeout. `apply_commit_result()` handles CM DemotedToColdRemoteTr response. `mature_peers()` promotes fresh duplex peers past 15-min threshold. `update_responder_counters()` derives IG events from counter changes. `verify_remote_transition()` validates upstream transition table. 43 tests.
- Diffusion-layer types (`diffusion.rs`): pure type module for upstream `Ouroboros.Network.Mux`, `ConnectionHandler`, and `Server2` types. `TemperatureBundle<T>` (hot/warm/established) with map/iter. `ProtocolTemperature`. `MiniProtocolStart` (StartEagerly/StartOnDemand/StartOnDemandAny). `MiniProtocolLimits` + `MiniProtocolDescriptor`. `OuroborosBundle` = `TemperatureBundle<Vec<MiniProtocolDescriptor>>`. Bundle constructors: `ntn_ouroboros_bundle()`, `ntc_ouroboros_bundle()`. `ControlMessage` (Continue/Quiesce/Terminate). `MuxMode` (Initiator/Responder/InitiatorResponder). `RateLimitDecision` with `rate_limit_decision()`. Error policy types (`ErrorCommand`, `RethrowPolicy`, `ErrorPolicyResult`). `PeerConnectionHandle` (per-connection governor→runtime bridge). `PeerStateAction` (Establish/Activate/Deactivate/Close). `RepromoteDelay`. 28 tests.
- Peer-state action bridge module (`peer_state_actions.rs`) is now present as a crate-owned seam between pure governor decisions and effectful runtime execution. It provides `PeerStateActions` runtime executor trait plus `governor_action_to_peer_state_action()` / `governor_actions_to_peer_state_actions()` translation helpers, preserving pure decision logic in `crates/network` while preparing runtime integration in `node`.
- `FetchMode` is now a single unified type: `crates/network::governor::FetchMode` (variants `FetchModeBulkSync` / `FetchModeDeadline`) is the only definition, and `crates/network::blockfetch_pool` re-exports it via `pub use crate::governor::FetchMode`. The previous duplicate `blockfetch_pool::FetchMode { BulkSync, Deadline }` has been deleted; its per-peer concurrency-cap logic now lives in `blockfetch_pool::max_concurrency_per_peer(mode)` (a free function rather than an inherent method, since the enum is owned by the governor module). The unification is pinned by `fetch_mode_is_unified_with_governor_module` (compares `TypeId::of::<blockfetch_pool::FetchMode>()` to `TypeId::of::<governor::FetchMode>()`) so a future regression that re-introduces a duplicate enum fails CI cleanly. The runtime governor tick (`node/src/runtime.rs::run_governor_loop`) now also propagates the per-tick `fetch_mode_from_judgement(LedgerStateJudgement)` signal into the shared `BlockFetchPool` via `BlockFetchPool::set_mode`, so the per-peer concurrency cap actually tracks the live ledger judgement instead of staying pinned to whatever mode the pool was constructed with. Reference: upstream `Ouroboros.Network.BlockFetch.ConsensusInterface.FetchMode` and `mkReadFetchMode` are also a single canonical type used by both the consensus interface and the BlockFetch decision policy (`bfcMaxConcurrency{BulkSync,Deadline}`).
