### Companion fix: parallel-test isolation in `local_server::tests`

Re-running `cargo test-all` after the toolchain provision and pin
refresh surfaced an intermittent failure:

```text
test local_server::tests::effective_era_index_pv_table_matches_upstream ... FAILED

assertion `left == right` failed:
  PV Some((1, 0)) should map to era_index 0, got 6
    left:  6
    right: 0
```

Running the test in isolation passed. Investigation: the
table-pinning test reads `YGG_LSQ_ERA_FLOOR` indirectly via
`effective_era_index_for_lsq` (which floors the reported era to the
configured value) but did not hold the function-local `ENV_LOCK`
defined inside the sister `era_floor_env_var_promotes_reported_era`
test. When Rust's test runner ran them in parallel, the floor-promotion
test could leak `YGG_LSQ_ERA_FLOOR=6` mid-flight and the table test
saw every PV (including Byron `(1, 0)`) mapped to era_index `6`
(Conway).

Fix in `node/src/local_server.rs`:

- Lifted `static ENV_LOCK: Mutex<()> = Mutex::new(());` from inside
  `era_floor_env_var_promotes_reported_era` to the test-module scope
  (immediately after the `use` statements, before the first `#[test]`).
- `effective_era_index_pv_table_matches_upstream` now acquires the
  same `ENV_LOCK` and calls `std::env::remove_var("YGG_LSQ_ERA_FLOOR")`
  before the assertions, so any value left over from a concurrent
  test cannot reach the floor.
- `era_floor_env_var_promotes_reported_era` now uses the module-level
  static instead of the function-local one (no behavior change in
  isolation; eliminates the duplicate static).

### Bounded preprod BlockFetch soak — discovered parity gap at slot ~429460

To exercise the §6.5 multi-peer BlockFetch harness end-to-end on the
freshly built R249 binary, a short bounded soak was run against the
preprod bootstrap peer:

```sh
node/dev/evidence/parallel_blockfetch_soak.sh \
  NETWORK=preprod \
  MAX_CONCURRENT_BLOCK_FETCH_PEERS=2 \
  EXPECT_WORKERS=1 \
  RUN_SECONDS=180 \
  SAMPLE_INTERVAL_S=30
```

`EXPECT_WORKERS=1` matches the single `bootstrapPeer` in
`node/configuration/preprod/topology.json` (the pre-`useLedgerAfterSlot`
window means ledger-derived peers do not populate the registry yet).
The MAX_CONCURRENT_BLOCK_FETCH_PEERS=2 knob still exercises the
multi-peer dispatch code path; reaching ≥2 simultaneously registered
workers requires either expanded `localRoots` or sync past
`useLedgerAfterSlot=112406400`.

#### Soak result

**The smoke test surfaced a previously-unobserved parity gap.** The
node started cleanly, registered the bootstrap peer as a worker, and
synced from `Origin` through 4 epoch boundaries (Byron → Shelley
transition observed via 4 `Ledger.EpochBoundary` notices, all with
`pparamUpdatesApplied=0`). It advanced past R207's previous preprod
evidence ceiling at slot `87440` and persisted ledger checkpoints at
`281460 → 286460 → … → 429460` before exiting:

```text
[1778005262145] Info Node.Recovery.Checkpoint slot=424460 (skipped)
[1778005263345] Info Node.Recovery.Checkpoint slot=429460 (skipped)
[1778005264045] Notice Net.Governor peer governor stopped
[1778005264045] Error Node.Sync node run failed
                error=consensus error: invalid VRF proof
                primaryPeer=3.79.79.217:3001
Error: consensus error: invalid VRF proof

Caused by: invalid VRF proof
Location:  node/src/main.rs:3625:32
```

#### Why this is a real parity gap, not an environmental issue

Preprod's `shelley-genesis.json` pins `decentralisationParam: 1`
(full federation) at Shelley genesis with `epochLength: 432000` and
`slotLength: 1`. The 4 observed epoch boundaries (`newEpoch=1..4`)
all reported `pparamUpdatesApplied=0`, so `d` stayed at `1` through
epoch 4. At `d = 1`, **every slot is a TPraos overlay slot** — the
same code path R248 (2026-05-02) was meant to close after the preview
slot `106220` reproduction.

The failing slot is therefore in the preprod Shelley federation
period (`epochLength * 4 = 345600 ≤ slot ≤ 432000`), past the end of
R207's previous coverage. The block was fetched from the official
preprod bootstrap peer (`3.79.79.217:3001`, IOG infrastructure) —
not a malicious or corrupt source — and Haskell `cardano-node`
synchronizes this region without complaint, so the divergence is
on the Yggdrasil side.

R248 already closed the preview-side TPraos active-overlay branch by
verifying the selected genesis delegate cold key and both VRF proofs
when the slot classifies as active overlay. Reserved non-active
overlay slots fail closed via `ConsensusError::InvalidVrfProof`. The
preprod failure shape strongly suggests one of the following edge
cases is in play:

1. `node/src/sync.rs::lookup_tpraos_overlay_schedule` returns
   `Reserved` for a slot upstream considers active (off-by-one or
   wrong cycle through the genesis delegation map), so we fail
   closed when we should have run the active branch.
2. The active-genesis-delegate VRF check itself fails on a real
   active slot — wrong VRF input construction, wrong seed/nonce
   tag, or wrong cold-key/VRF-key pair selection.
3. The TPraos nonce evolution drifts somewhere in the 4 epoch
   transitions so by epoch 4 our `epoch_nonce` no longer matches
   upstream's `Nonce` for that slot.

#### Discovered: Gap BP — Plutus V2 cost-budget parity overrun at slot ~1,462,057

After 16 clean epoch boundaries the node failed at preview slot
`~1,462,057` (well past R246's slot `901,725` refscan and R248's slot
`868,687` live-resume ceilings) with:

```text
ledger decode error: Phase-2 validation tag mismatch:
  block says true, re-evaluation says false:
  script 86f081bd6de5712f1bd1d8fe8a25fdb8782830522db18550c365d1df failed:
  purpose Spending { tx_id: 202f45d6…ed68, index: 0 },
  version V2,
  tx 7bb40e40c3e6010ead628fd9ea62ae4f8acab340ccca26efacad826caa3be5b9,
  ex_units(mem=6_121_408, steps=1_657_962_006):
  out of budget: 200 accumulated steps:
    cost cpu=4_600_000, mem=20_000,
    before cpu=4_293_691, mem=67_426,
    after cpu=-306_309, mem=47_426
```

#### Why this is a real parity bug

The on-chain block declared `is_valid=true`. Upstream Haskell node
synchronizes this region without complaint. Yggdrasil's CEK
re-evaluation overruns the declared `(mem=6_121_408,
steps=1_657_962_006)` budget by `cpu=306_309` (≈ 0.0185 % of the
total CPU budget) at one batched-step debit.

The trace format `200 accumulated steps: cost cpu=4_600_000,
mem=20_000` is `crates/plutus/src/machine.rs::spend_accumulated_step_budget`
charging the cumulative cost of `DEFAULT_STEP_SLIPPAGE = 200` CEK
steps batched together. The 4.6 M CPU debit at this point ÷ 200 steps
≈ 23 K CPU/step, matching expected per-step CPU cost order — so this
is not a single-builtin overcharge but a per-step cost-model rounding
that accumulates across the ~14.2 M total CEK steps the script
executes.

Plausible root causes (ordered by likelihood):

1. **Per-`StepKind` CPU cost off by a small constant** vs. upstream
   V2 cost-model B JSON. A 0.0185 % drift over 14.2 M steps would
   show up as 23 K × ε per step. `crates/plutus/src/cost_model.rs`'s
   `step_cost(kind)` table needs to be cross-checked against
   `plutus-core/cost-model/data/builtinCostModelB.json`'s
   `cek*ConstCost`/`cekVarCost`/etc. fields.
2. **Step-kind miscount on a specific Plutus V2 term shape**, e.g.
   a Force/Delay path that increments two step kinds where upstream
   increments one.
3. **Cost-model parameter rounding** in our parser between the JSON
   integer constants and our internal `StepCost` values.

V2 scripts do not exercise `Constr`/`Case` (V3-only SIR), so those
step kinds are not the culprit; the divergence has to be in the V2
step kinds (`Constant`, `Var`, `Lambda`, `Apply`, `Delay`, `Force`,
`Builtin`).

#### Sync-only escape hatch shipped in this slice

To unblock chain catch-up while Gap BP is investigated, this slice
adds the `YGG_SKIP_PHASE2` env var. When set, the new
`node/src/sync.rs::phase2_evaluator_or_trust_block` helper passes
`None` to `apply_block_validated` at all 6 production call sites
(epoch-boundary-aware advance, fallback advance, recovery, and the
initial-sync rollback-to-Origin reapply path), so the ledger applies
each transaction based on its on-chain `is_valid` tag without
re-running Plutus scripts.

Phase-1 validation remains in force in trust mode: fees
(`validate_fee` / `validate_conway_fee`), Ed25519 witnesses, UTxO
state transitions, signature verification, protocol-parameter
update collection and application, certificate processing,
withdrawal handling. The block author's `is_valid` tag is trusted
because the block was already accepted by the network majority.

The helper emits a one-time stderr `WARN: YGG_SKIP_PHASE2 is set`
trace via an `AtomicBool::compare_exchange` guard so operators
see they are running in trust mode. Setting the env var on a
block-producing node would silently mask CEK divergence at forge
time, so the doc-comment explicitly forbids that use.
