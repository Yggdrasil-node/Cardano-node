## Functional shippable surface

After R430, `cargo run --bin cardano-tracer -- -c <config>` boots
a real Acceptors supervisor that:

1. Reads + validates the operator's `tracer-config.json`.
2. Decides between server-mode (`AcceptAt`) and client-mode
   (`ConnectTo`).
3. Binds a `LocalPeerListener` (server) or dials via
   `UnixStream::connect` (client).
4. Initializes per-connection mux + spawns the trace-objects
   sub-protocol acceptor driver.
5. On graceful shutdown, drains in-flight `MsgDone` handshakes
   within a 15-second budget.

The per-node Prometheus + EKG-equivalent HTTP endpoints from
R408-R414 are wired against a real `MetricsStore` registry — once
trace-forwarder ingest is operationally exercised against a
running cardano-node forwarder (post-handshake-codec port), the
endpoints will serve real metrics data.

## What ships in follow-on arcs

The R411-R430 arc is **structurally** complete; full operational
trace-forwarder ingest needs:

1. **Trace-forwarder handshake codec port** — the upstream
   `forwardingVersionCodec` from
   `Trace.Forward.Utils.Version.hs`. Required for RemoteSocket
   TCP path + same-host trust verification on LocalPipe.
2. **TraceObject CBOR codec** — the `Cardano.Logging.TraceObject`
   `Serialise` instance from `trace-dispatcher`. R424's stub
   decoder returns empty list; real ingest needs the full codec.
3. **Cardano-node forwarder side** — the trace-forwarder
   *initiator* on the node side (mirrors of `Trace.Forward.Run.
   TraceObject.Forwarder` + the cardano-node integration). This
   is a separate parity arc tracked under `node/` rather than
   `crates/cardano-tracer/`.

Once those three follow-on arcs land, an operator running both
yggdrasil cardano-tracer and yggdrasil cardano-node will have
full byte-equivalent trace-forwarder traffic relative to upstream
binaries.
